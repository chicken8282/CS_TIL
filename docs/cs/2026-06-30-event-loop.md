# 이벤트 루프(Event Loop)

- **비동기 작업을 싱글 스레드에서 처리하게 해주는 실행 모델**이다.
- **콜 스택, 태스크 큐, 마이크로태스크 큐**의 흐름을 이해해야 한다.
- 면접에서는 **setTimeout, Promise, async/await 실행 순서**로 자주 묻는다.

## 개념 설명

이벤트 루프는 자바스크립트 같은 싱글 스레드 환경에서 **동시에 여러 작업이 돌아가는 것처럼 보이게 하는 메커니즘**이다.  
실제 실행은 한 번에 하나의 콜 스택(Call Stack)에서만 일어난다. 그런데 네트워크 요청, 타이머, DOM 이벤트처럼 오래 걸리는 작업은 브라우저나 런타임이 맡아 처리하고, 완료되면 큐에 넣는다.

핵심 흐름은 다음과 같다.

1. **콜 스택**: 현재 실행 중인 함수가 쌓인다.
2. **태스크 큐(매크로태스크 큐)**: `setTimeout`, 이벤트 콜백 같은 작업이 대기한다.
3. **마이크로태스크 큐**: `Promise.then`, `queueMicrotask` 같은 작업이 대기한다.
4. **이벤트 루프**: 콜 스택이 비면 먼저 **마이크로태스크 큐**를 모두 비우고, 그다음 태스크 큐에서 하나를 꺼내 실행한다.

즉, 실행 순서는 보통  
**동기 코드 → 마이크로태스크 → 태스크**  
순으로 이해하면 된다.  
`async/await`는 내부적으로 Promise 기반이므로, `await` 뒤의 코드는 마이크로태스크처럼 동작하는 경우가 많다.

## 짧은 예시/비유

비유하면 **콜 스택은 현재 처리 중인 주문 1건**, 큐는 **대기표**다.  
주문을 하나 처리하는 동안 다른 요청은 대기표에 쌓이고, 현재 주문이 끝나면 먼저 **급한 메모(Promise)** 를 처리한 뒤, 일반 대기표(`setTimeout`)를 처리한다.

```js
console.log("A");

setTimeout(() => console.log("B"), 0);

Promise.resolve().then(() => console.log("C"));

console.log("D");
```

실행 순서: **A → D → C → B**  
이유: 동기 코드가 먼저 끝나고, Promise가 태스크보다 우선 처리되기 때문이다.

## 면접 질문 2개

**Q1. `setTimeout(fn, 0)`은 왜 즉시 실행되지 않나요?**  
A. 최소 지연 시간 후 태스크 큐에 들어가며, 콜 스택이 비어야 실행되기 때문이다.

**Q2. Promise가 `setTimeout`보다 먼저 실행되는 이유는?**  
A. Promise의 `.then()`은 마이크로태스크 큐에 들어가고, 이벤트 루프가 태스크 큐보다 마이크로태스크 큐를 먼저 비우기 때문이다.

## 한 줄 정리

**이벤트 루프는 싱글 스레드에서도 동기·비동기 작업의 실행 순서를 조율하는 핵심 메커니즘이다.**
