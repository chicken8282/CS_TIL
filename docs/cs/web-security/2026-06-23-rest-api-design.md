# REST API 설계 핵심 정리

- **리소스 중심 설계**: URI는 동작이 아니라 대상(명사)을 표현한다.
- **HTTP 메서드 의미 준수**: GET/POST/PUT/PATCH/DELETE를 역할에 맞게 사용한다.
- **일관성과 상태코드**: 응답 형식, 에러 규칙, 상태코드를 통일해야 유지보수가 쉽다.

## 개념 설명

REST API는 웹의 HTTP를 활용해 **리소스(Resource)** 를 표준적으로 다루는 설계 방식이다. 핵심은 “무엇을 할지”보다 “무엇을 다룰지”를 먼저 정하는 것이다. 예를 들어 사용자를 조회한다면 `/users/1`처럼 리소스를 나타내고, 조회는 `GET`, 생성은 `POST`, 전체 수정은 `PUT`, 부분 수정은 `PATCH`, 삭제는 `DELETE`를 사용한다.

좋은 REST API는 URI가 직관적이고 예측 가능해야 한다. `/getUser`, `/deleteUser`처럼 동사를 넣기보다 `/users`, `/users/{id}`처럼 설계한다. 또한 상태코드도 중요하다. 성공은 `200`, 생성은 `201`, 잘못된 요청은 `400`, 인증 실패는 `401`, 권한 없음은 `403`, 찾을 수 없음은 `404`를 사용하면 클라이언트가 분기 처리를 쉽게 할 수 있다.

실무에서는 **버전 관리**, **에러 응답 포맷 통일**, **페이징/정렬/필터링 규칙**까지 포함해야 한다. 예: `/users?page=1&size=20&sort=name,asc`.

## 짧은 예시 / 비유

REST API는 **도서관 시스템**과 비슷하다.  
- 책(리소스): `/books/10`  
- 책 조회: `GET /books/10`  
- 새 책 등록: `POST /books`  
- 책 정보 수정: `PATCH /books/10`  
- 책 폐기: `DELETE /books/10`  

즉, “책을 가져와라/수정해라”가 아니라 “책이라는 대상”을 정하고, 그 대상에 대한 작업은 HTTP 메서드가 담당한다.

## 면접 질문

**Q1. RESTful 하다는 것은 무엇인가요?**  
A. URI는 리소스를 표현하고, HTTP 메서드와 상태코드를 의미에 맞게 사용하며, 일관된 인터페이스를 지키는 것을 말합니다.

**Q2. PUT과 PATCH의 차이는 무엇인가요?**  
A. `PUT`은 보통 리소스 전체를 교체하는 의미이고, `PATCH`는 일부 속성만 부분 수정하는 데 사용합니다.

## 한 줄 요약

REST API 설계는 **리소스 중심, HTTP 의미 준수, 일관성 유지**가 핵심이다.
