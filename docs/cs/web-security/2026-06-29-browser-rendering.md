# 브라우저 렌더링

- **렌더링은 HTML/CSS/JS를 화면 픽셀로 바꾸는 과정**이다.
- **DOM, CSSOM, 렌더 트리, 레이아웃, 페인트, 합성** 순으로 진행된다.
- **성능 최적화의 핵심은 재계산과 재배치를 줄이는 것**이다.

## 개념 설명

브라우저 렌더링은 사용자가 입력한 URL의 문서를 화면에 그리는 전체 과정이다. 먼저 HTML을 파싱해 **DOM 트리**를 만들고, CSS를 파싱해 **CSSOM 트리**를 만든다. 이 둘이 결합되어 **렌더 트리**가 생성되며, 화면에 보이는 요소만 포함된다.

이후 각 요소의 크기와 위치를 계산하는 **레이아웃(Layout/Reflow)** 이 수행된다. 예를 들어 글자 크기, 박스 크기, 화면 비율에 따라 각 요소가 어디에 배치될지 계산한다. 다음으로 색, 그림자, 테두리 같은 시각 정보를 칠하는 **페인트(Paint)** 단계가 있고, 마지막으로 레이어 단위로 화면에 합쳐지는 **합성(Composite)** 이 진행된다.

JavaScript는 렌더링 중간에도 DOM과 스타일을 변경할 수 있다. 이때 `width`, `height`, `offsetHeight` 같은 값을 읽으면 레이아웃 계산이 강제로 일어날 수 있다. 따라서 DOM 변경과 읽기를 섞어 쓰면 성능이 나빠질 수 있다. 실무에서는 `transform`, `opacity`처럼 합성만 유도하는 속성을 활용하면 렌더링 부담을 줄이기 좋다.

## 짧은 예시/비유

렌더링은 **집을 짓는 과정**과 비슷하다.  
- DOM: 설계도  
- CSSOM: 인테리어 규칙  
- 레이아웃: 방의 배치 결정  
- 페인트: 벽 칠하기  
- 합성: 완성된 집을 한 장의 사진으로 보여주기  

예를 들어 버튼의 `width`를 바꾸면 방 배치를 다시 해야 하므로 레이아웃이 다시 돈다. 반면 `opacity`를 바꾸면 색만 바뀌는 경우가 많아 상대적으로 비용이 적다.

## 면접 질문 2개

**Q1. 렌더 트리와 DOM의 차이는?**  
A. DOM은 문서 전체 구조이고, 렌더 트리는 그중 화면에 보이는 요소만 포함한 구조다.

**Q2. 리플로우와 리페인트의 차이는?**  
A. 리플로우는 위치·크기 재계산, 리페인트는 색·스타일 다시 그리기다. 리플로우가 더 비싸다.

## 한 줄 요약

**브라우저 렌더링은 DOM/CSSOM에서 시작해 레이아웃·페인트·합성으로 끝나며, 성능은 재계산을 얼마나 줄이느냐에 달려 있다.**
