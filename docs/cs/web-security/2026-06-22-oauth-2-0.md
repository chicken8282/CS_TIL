# OAuth 2.0

- **인증(Authentication)**이 아니라 **권한 부여(Authorization)** 표준이다.
- 비밀번호를 서비스 간에 직접 공유하지 않고, **토큰**으로 접근 권한을 위임한다.
- 대표 흐름은 **Authorization Code Flow**이며, 보안상 가장 많이 사용된다.

## 개념 설명

OAuth 2.0은 사용자가 자신의 계정 자원에 대해, 제3자 애플리케이션이 **일부 기능만 접근**하도록 허용하는 프로토콜이다. 예를 들어 “구글 계정으로 로그인”처럼 보이지만, 핵심은 사용자의 구글 비밀번호를 앱에 넘기는 것이 아니라 **구글이 발급한 액세스 토큰**으로 API 접근을 허용하는 것이다.

주요 구성 요소는 다음과 같다.
- **Resource Owner**: 사용자
- **Client**: 권한을 요청하는 애플리케이션
- **Authorization Server**: 사용자 동의 후 토큰 발급
- **Resource Server**: 토큰을 검증하고 자원 제공

토큰 종류도 중요하다.
- **Access Token**: 짧은 수명, API 호출에 사용
- **Refresh Token**: Access Token 재발급용, 더 길게 유지

실무에서는 **Redirect URI 검증**, **PKCE**, **State 값 검증**이 중요하다. 특히 모바일/SPA 환경에서는 Authorization Code Flow + PKCE를 써서 코드 탈취 위험을 줄인다.

## 짧은 예시 또는 비유

호텔 키카드로 생각하면 쉽다.  
비밀번호는 호텔 프런트에만 있고, 손님은 객실 출입용 **키카드(토큰)**만 받는다. 키카드는 정해진 방과 시간에만 유효하다. 즉, 비밀번호를 공유하지 않고 필요한 권한만 잠시 위임하는 구조다.

## 면접 질문 2개

**Q1. OAuth 2.0은 인증인가요, 인가인가요?**  
A. 기본적으로 **인가** 표준이다. 로그인처럼 보일 수 있지만, 핵심은 자원 접근 권한 위임이다.

**Q2. Access Token과 Refresh Token의 차이는?**  
A. Access Token은 API 호출용의 짧은 수명 토큰이고, Refresh Token은 만료된 Access Token을 다시 발급받기 위한 토큰이다.

**한 줄 요약:** OAuth 2.0은 비밀번호를 넘기지 않고, 토큰으로 제3자에게 안전하게 접근 권한을 위임하는 표준이다.
