# LRU Cache: 가장 최근에 쓴 것만 남기는 캐시

- **핵심 아이디어**: 최근 사용한 데이터는 오래 살아남고, 오래 안 쓴 데이터는 먼저 제거한다.
- **구현 포인트**: `O(1)` 조회와 갱신을 위해 **해시맵 + 이중 연결 리스트** 조합을 쓴다.
- **실수 포인트**: 노드 이동, 삭제, 삽입 시 포인터 갱신 누락이 가장 흔한 버그다.

## 개념 설명

LRU(Least Recently Used) 캐시는 **가장 오랫동안 사용되지 않은 항목을 제거**하는 정책이다. 예를 들어 캐시 용량이 3일 때 `A, B, C`를 넣고 `A`를 다시 접근하면, 사용 순서는 `B, C, A`가 된다. 이후 `D`가 들어오면 가장 오래된 `B`가 제거된다.

실무 구현에서는:
- **해시맵**으로 key → 노드 위치를 빠르게 찾고
- **이중 연결 리스트**로 최근 사용 순서를 유지한다.

조회(`get`) 시 해당 노드를 리스트 앞으로 옮기고, 삽입(`put`) 시 새 노드를 맨 앞에 넣는다. 용량 초과 시 뒤쪽 노드를 삭제한다. 이 방식으로 평균 `O(1)`을 달성한다.

### 자주 하는 실수 / 안티패턴
- **배열만으로 구현**: 중간 삭제/이동이 `O(n)`이라 성능이 망가진다.
- **해시맵만 사용**: 순서를 유지할 수 없어 LRU 정책을 구현하기 어렵다.
- **더미 헤드/테일 미사용**: 경계 조건 처리 코드가 복잡해지고 버그가 늘어난다.
- **접근 시 순서 갱신 누락**: `get`도 “사용”이므로 반드시 최근 위치로 옮겨야 한다.
- **용량 초과 후 맵에서 미삭제**: 리스트에서만 지우면 해시맵에 죽은 참조가 남는다.

## 짧은 예시 / 비유

비유하면 책상 위 메모 3장이다. 새 메모를 쓰거나 다시 보면 가장 위에 올려둔다. 공간이 부족하면 맨 아래, 즉 가장 오래 안 본 메모를 버린다.  
예: `put(1), put(2), get(1), put(3)` → `2`가 가장 오래 안 써서 제거된다.

## 면접 질문 2개

**Q1. LRU를 왜 해시맵과 이중 연결 리스트로 구현하나요?**  
A. 조회는 해시맵으로 `O(1)`, 순서 갱신과 삭제는 이중 연결 리스트로 `O(1)`에 처리하기 위해서다.

**Q2. `get()`도 리스트 순서를 바꿔야 하나요?**  
A. 그렇다. `get()`은 “최근 사용”이므로 해당 노드를 앞으로 이동하지 않으면 LRU 정책이 깨진다.

## 한 줄 요약

**LRU는 “최근 사용 여부”를 정확히 반영해야 하며, 해시맵 + 이중 연결 리스트에서 포인터/맵 동기화가 핵심이다.**
