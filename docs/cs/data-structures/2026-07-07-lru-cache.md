# LRU Cache

- **가장 최근에 사용한 항목만 살리고, 오래 안 쓴 항목은 제거**하는 캐시 정책이다.
- 구현 핵심은 보통 **해시맵 + 이중 연결 리스트** 조합이다.
- 실무/면접에서 자주 틀리는 포인트는 **갱신 순서, O(1) 보장, 동시성**이다.

## 개념 설명

LRU(Least Recently Used) 캐시는 **마지막 사용 시점이 가장 오래된 데이터부터 삭제**한다.  
예를 들어 용량이 3일 때 `A, B, C`가 들어 있고 `B`를 다시 접근하면 순서는 `A, C, B`처럼 바뀐다. 이후 `D`가 들어오면 가장 오래된 `A`가 제거된다.

대표 구현은 다음과 같다.

- **해시맵**: 키로 노드를 바로 찾는다. 조회를 O(1)로 만든다.
- **이중 연결 리스트**: 사용 순서를 관리한다. 최근 사용 항목을 맨 뒤로 이동시킨다.

### 자주 하는 실수 / 안티패턴
- **리스트만 사용**: 탐색이 O(n)이 되어 캐시 의미가 약해진다.
- **접근 시 순서 갱신 누락**: `get()` 했는데도 최근 사용으로 이동하지 않으면 LRU가 아니다.
- **삽입/삭제 후 맵과 리스트 동기화 실패**: 삭제된 노드가 맵에 남아 있으면 버그가 난다.
- **용량 초과 시 잘못된 노드 제거**: 머리/꼬리 포인터를 헷갈려 오래된 항목 대신 최근 항목을 지우는 경우가 많다.
- **스레드 안전성 무시**: 멀티스레드 환경에서는 race condition으로 순서가 깨질 수 있다.

## 짧은 예시 / 비유

비유하자면, 책상 위에 놓는 **최대 3권의 자주 보는 책**이다.  
최근 본 책은 가장 손 닿기 쉬운 곳에 두고, 한동안 안 본 책부터 치운다.  
즉, “마지막으로 쓴 것”을 기억하는 정리 규칙이다.

## 면접 질문 2개

**Q1. LRU 캐시를 O(1)로 구현하려면?**  
A1. 키 조회는 해시맵, 순서 관리는 이중 연결 리스트를 사용한다.

**Q2. `get()` 연산에서 왜 노드를 이동시켜야 하나?**  
A2. 조회 자체가 “최근 사용”이기 때문이다. 이동하지 않으면 정책이 깨진다.

## 한 줄 정리

**LRU는 “자주가 아니라 최근”을 기준으로, 해시맵 + 이중 연결 리스트로 O(1) 관리하는 캐시다.**
