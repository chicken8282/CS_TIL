# 시스템 콜(System Call)

- **유저 모드에서 커널 모드로 들어가는 공식 경로**다.
- **파일 I/O, 네트워크, 프로세스 제어** 같은 OS 자원을 쓰려면 결국 시스템 콜을 거친다.
- **실무 트러블슈팅의 핵심 단서**는 `errno`, 지연 시간, 블로킹 여부, 권한/FD 상태다.

## 개념 설명

시스템 콜은 사용자 프로그램이 운영체제 기능을 요청하는 인터페이스다. 예를 들어 `open()`, `read()`, `write()`, `fork()`, `execve()`, `socket()` 같은 호출은 직접 하드웨어를 만지는 것이 아니라 커널에 작업을 위임한다.  
중요한 점은 **컨텍스트 전환 비용**이 존재한다는 것이다. 자주 호출하면 성능이 떨어질 수 있어, 실무에서는 작은 `write()`를 여러 번 하기보다 버퍼링을 사용한다.

트러블슈팅에서는 다음을 먼저 본다:
- `-1` 반환과 `errno` 값
- `EINTR`, `EAGAIN`, `EBADF`, `EACCES` 같은 대표 에러
- 블로킹/논블로킹 설정 여부
- 파일 디스크립터(fd) 누수, 권한, 파일 시스템 상태
- 네트워크 계열에서는 타임아웃과 재시도 정책

예: 로그가 안 찍히면 단순 코드 문제가 아니라 `write()`가 실패했는지, 디스크가 꽉 찼는지, FD가 닫혔는지부터 확인해야 한다.

## 짧은 예시 / 비유

시스템 콜은 **회사에서 총무팀에 공식 요청서를 넣는 것**과 비슷하다.  
직접 창고에 들어가 물건을 꺼내면 안 되고, 정해진 양식으로 요청해야 처리된다.  
예를 들어 `read(fd, buf, 1024)`는 “이 파일에서 1024바이트 읽어줘”라는 요청이고, 커널이 권한과 상태를 확인한 뒤 데이터를 전달한다.

## 면접 질문 2개

**Q1. 시스템 콜과 라이브러리 함수의 차이는?**  
A. 라이브러리 함수는 사용자 공간에서 동작할 수 있고, 시스템 콜은 커널로 진입한다. `printf()`는 내부에서 `write()`를 호출할 수 있다.

**Q2. 실무에서 시스템 콜 실패를 어떻게 디버깅하나?**  
A. 반환값과 `errno`를 확인하고, `strace`, `lsof`, `ulimit`, 로그/권한/FD 상태를 함께 본다.

## 한 줄 요약

시스템 콜은 OS 자원 접근의 관문이며, 실무에서는 **성능보다 실패 원인과 상태 확인**이 더 중요하다.
