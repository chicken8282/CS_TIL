# IPC(Inter-Process Communication)

- 서로 다른 프로세스는 **메모리 주소 공간이 분리**되어 있어 직접 데이터를 읽지 못한다.
- IPC는 **커널을 경유한 데이터 전달**과 **동기화**를 통해 프로세스 간 협업을 가능하게 한다.
- 구현 관점에서 핵심은 **복사 비용, 블로킹/논블로킹, 동기화 원자성**이다.

## 개념 설명

IPC는 프로세스 간 통신을 위한 메커니즘이다. 내부적으로는 보통 **커널이 중간 저장소** 역할을 한다. 예를 들어 파이프는 커널 버퍼에 바이트 스트림을 저장하고, 송신 프로세스는 `write()`, 수신 프로세스는 `read()`로 접근한다. 이때 유저 공간 → 커널 공간 → 유저 공간으로 데이터가 이동하므로 **컨텍스트 스위치와 메모리 복사 비용**이 발생한다.

대표 방식은 다음과 같다.

- **Pipe / Named Pipe(FIFO)**: 단방향 바이트 스트림. 부모-자식 프로세스 통신에 자주 사용.
- **Message Queue**: 메시지 단위로 저장되며, 커널이 큐를 관리해 순서와 경계를 보존.
- **Shared Memory**: 가장 빠른 편. 여러 프로세스가 같은 물리 메모리 페이지를 매핑하므로 복사가 적다. 대신 **뮤텍스/세마포어**로 동기화해야 한다.
- **Socket**: 로컬/원격 모두 가능. 내부적으로는 소켓 버퍼와 프로토콜 스택을 통해 송수신한다.
- **Signal**: 작은 이벤트 알림용. 데이터 전달보다는 상태 통지에 적합하다.

실무 관점에서 중요한 점은 **동기화와 경계 조건**이다. 예를 들어 shared memory는 빠르지만, 동시에 읽고 쓰면 레이스 컨디션이 생긴다. 반면 pipe나 queue는 커널이 순서를 어느 정도 보장하지만, 버퍼가 가득 차면 송신이 블로킹될 수 있다.

## 짧은 예시 / 비유

비유하자면, IPC는 **직접 말할 수 없는 두 사무실이 공용 우편함을 쓰는 것**과 같다.  
- pipe: 한 방향 쪽지  
- message queue: 번호표가 있는 우편함  
- shared memory: 공동 작업 테이블  
- socket: 같은 건물/다른 건물 모두 연결 가능한 전화선

예시: 부모 프로세스가 자식 프로세스에 작업 명령을 보내고, 자식은 결과를 pipe로 반환한다. 이때 부모가 너무 빨리 쓰면 pipe 버퍼가 차서 `write()`가 막힐 수 있다.

## 면접 질문 2개

**Q1. shared memory가 가장 빠른 이유는?**  
A. 커널을 통한 반복 복사가 적고, 같은 메모리 페이지를 매핑해 직접 읽기 때문이다.

**Q2. pipe와 socket의 차이는?**  
A. pipe는 보통 같은 호스트 내부의 단방향 바이트 스트림이고, socket은 로컬/네트워크 모두 가능하며 양방향 통신과 프로토콜 계층을 지원한다.

**한 줄 요약:** IPC는 프로세스 분리라는 제약을 커널, 버퍼, 동기화로 우회하는 통신 기술이다.
