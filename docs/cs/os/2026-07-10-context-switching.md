# 컨텍스트 스위칭(Context Switching)

- **정의**: CPU가 실행 중인 프로세스/스레드의 상태를 저장하고 다른 작업의 상태를 복원하는 과정
- **핵심 비용**: 문맥 저장·복원, 캐시/TLB 미스, 스케줄링 오버헤드로 인해 “일은 안 하는데 시간은 쓰는” 구간 발생
- **현업 포인트**: 동시성 높이려고 스레드를 무작정 늘리면 처리량이 오히려 떨어질 수 있음

## 개념 설명
컨텍스트 스위칭은 OS 스케줄러가 CPU를 다른 작업에 배분하기 위해 현재 실행 중인 작업의 레지스터, PC(프로그램 카운터), 스택 정보 등을 저장하고, 다음 작업의 상태를 복구하는 과정이다. 프로세스 간 전환은 주소 공간도 바뀌므로 스레드 전환보다 일반적으로 더 무겁다.  
현업에서는 **웹 서버의 스레드 풀 크기**, **배치 작업의 병렬도**, **실시간 이벤트 처리 파이프라인**에서 컨텍스트 스위칭 비용을 직접 고려한다. 예를 들어 요청당 스레드를 하나씩 늘리면 대기 시간은 줄어들 수 있지만, 동시 스레드가 과도해지면 CPU가 실제 계산보다 스케줄링과 캐시 재적재에 시간을 더 쓰게 된다. 결과적으로 응답 지연이 늘고, p99 latency가 악화될 수 있다.

## 짧은 예시/비유
요리사 1명이 동시에 10개 요리를 번갈아 만드는 상황을 생각하자. 각 요리의 조리 상태를 메모하고 도구를 바꿔 끼우는 시간이 계속 든다. 요리사가 10개를 “병렬”로 하는 것 같지만 실제로는 전환 비용 때문에 전체 완료 시간이 늘어날 수 있다.

## 면접 질문
**Q1. 컨텍스트 스위칭이 왜 비싼가요?**  
A. 상태 저장/복원 외에도 캐시 지역성이 깨지고, TLB 및 분기 예측 효율이 떨어져 CPU 효율이 감소하기 때문입니다.

**Q2. 스레드를 많이 만들면 항상 성능이 좋아지나요?**  
A. 아닙니다. I/O 대기에는 유리할 수 있지만, CPU 바운드 작업에서는 컨텍스트 스위칭 증가로 오히려 느려질 수 있습니다.

**한 줄 요약**: 컨텍스트 스위칭은 동시성을 가능하게 하지만, 과하면 성능을 갉아먹는 숨은 비용이다.
