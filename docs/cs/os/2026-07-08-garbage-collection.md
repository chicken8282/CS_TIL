# 가비지 컬렉션(Garbage Collection)

- **GC는 메모리를 자동 해제**하지만, 언제 실행될지는 **예측할 수 없다**
- **자주 하는 실수**는 “GC가 알아서 다 해결”한다고 믿고 **객체 생명주기 관리**를 소홀히 하는 것
- **안티패턴**은 메모리 누수, 과도한 객체 생성, 불필요한 참조 유지다

## 개념 설명

가비지 컬렉션은 더 이상 프로그램에서 참조되지 않는 객체를 자동으로 찾아 메모리를 회수하는 메커니즘이다. Java, Go, Python 같은 언어에서 개발자가 `free()`를 직접 호출하지 않아도 되는 이유다.  
하지만 **자동 해제 = 메모리 관리가 필요 없다**는 뜻은 아니다. 오히려 GC 환경에서는 **참조를 끊는 습관**이 중요하다. 참조가 남아 있으면 객체는 살아 있다고 판단되어 회수되지 않는다.

### 자주 하는 실수와 안티패턴
- **컬렉션에 넣어둔 객체를 삭제하지 않음** → 캐시, 전역 리스트, 싱글톤에 쌓여 메모리 누수 발생
- **이벤트 리스너/콜백 해제 안 함** → 화면이 닫혀도 객체가 계속 참조됨
- **큰 객체를 지역 변수로 오래 잡아둠** → 필요 없는데도 GC 대상이 되지 않음
- **`System.gc()`에 의존** → 강제 호출은 보장도 없고 성능 예측도 어렵다
- **짧은 생명주기 객체를 과도하게 생성** → GC 빈도가 늘어 성능 저하

GC는 보통 도달 가능성(Reachability)을 기준으로 판단한다. 즉, 루트(root)에서 참조되는 객체는 살아 있고, 더 이상 도달할 수 없는 객체만 회수 대상이다.

## 짧은 예시 / 비유

비유하면, GC는 **창고 정리 직원**이다. 하지만 “버릴 물건”에 빨간 딱지를 붙여주는 건 개발자다.  
박스를 전부 전선으로 묶어두면 창고가 꽉 차도 직원이 못 치운다. 이게 바로 **참조가 남아 있는 메모리 누수**다.

```java
List<User> cache = new ArrayList<>();
cache.add(user);
// 나중에 더 이상 필요 없으면 remove 또는 null 처리
cache.remove(user);
```

## 면접 질문 2개

**Q1. GC가 있는데도 메모리 누수가 생기는 이유는?**  
A. 객체가 더 이상 사용되지 않아도 참조가 남아 있으면 GC가 회수하지 못하기 때문이다.

**Q2. `System.gc()`를 남발하면 왜 안 좋은가?**  
A. 실행 시점이 보장되지 않고, 강제 유도는 성능 저하와 멈춤 시간 증가를 유발할 수 있기 때문이다.

## 한 줄 요약

**GC는 자동화된 메모리 관리 도구지만, 참조를 끊는 설계는 여전히 개발자의 책임이다.**
