# Paging과 Segmentation

- **Paging**은 메모리를 고정 크기 페이지/프레임으로 나눠 **외부 단편화**를 줄이고, 주소 변환을 빠르게 한다.
- **Segmentation**은 코드/힙/스택 같은 **논리적 단위**로 나눠 보호와 공유가 쉽지만, **외부 단편화**가 생기기 쉽다.
- 성능 관점에서는 paging이 일반적으로 유리하고, 실제 시스템은 **세그먼트 개념을 논리 모델로 두고 paging으로 물리 배치**를 최적화한다.

## Concept explanation
Paging은 가상 주소를 `페이지 번호 + 오프셋`으로 분해하고, 페이지 테이블을 통해 물리 프레임을 찾는다. 페이지 크기가 작으면 메모리 활용률은 좋아지지만 페이지 테이블과 TLB 미스 비용이 커질 수 있다. 반대로 페이지 크기가 크면 관리 오버헤드는 줄지만 내부 단편화가 늘 수 있다. 그래서 성능 최적화 핵심은 **TLB 적중률 향상**, **지역성(locality) 활용**, **적절한 페이지 크기 선택**이다.

Segmentation은 프로그램 구조에 맞춰 주소를 `세그먼트 번호 + 오프셋`으로 처리한다. 코드/데이터/스택을 분리하므로 권한 제어와 공유가 자연스럽지만, 가변 크기라 빈 공간이 흩어지는 **외부 단편화**가 발생해 연속 할당 비용이 증가한다. 대규모 시스템에서는 이 단점이 성능 병목이 될 수 있어 paging이 더 널리 쓰인다.

실무/면접 포인트: “왜 paging이 주류인가?”에 대해 **단편화 감소, 스왑/할당 단순화, 하드웨어 지원(TLB)**을 함께 말하면 좋다.

```text
가상주소 = [page# | offset]
page# -> 페이지 테이블 -> frame#
물리주소 = [frame# | offset]
```

```mermaid
flowchart LR
A[가상 주소] --> B[페이지 번호/오프셋 분리]
B --> C[TLB 조회]
C -->|hit| G[물리 주소]
C -->|miss| D[페이지 테이블]
D --> E[프레임 번호]
E --> G
```

## Interview questions
1. **Paging이 Segmentation보다 성능상 유리한 이유는?**  
   외부 단편화가 적고, 고정 크기 관리로 할당/스왑이 단순하며 TLB 같은 하드웨어 최적화가 쉽다.

2. **Segmentation의 장점은 무엇인가?**  
   코드, 데이터, 스택을 논리적으로 분리해 보호와 공유 정책을 세밀하게 적용하기 좋다.

## Takeaway
**성능 최적화 관점에서는 paging이 메모리 관리의 표준이고, segmentation은 논리적 보호를 위한 개념적 도구로 이해하면 된다.**
