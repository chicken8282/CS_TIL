# Sharding과 Partitioning 정리

- **Partitioning**: 큰 데이터를 여러 조각으로 나눠 저장·처리하는 **상위 개념**이다.
- **Sharding**: Partitioning 중에서도 각 조각을 **서로 다른 서버/노드**에 분산하는 방식이다.
- 핵심은 **확장성(Scale-out)**, **성능**, **장애 격리**지만, **재분배 비용**과 **운영 복잡도**가 커진다.

## 개념 설명
Partitioning은 데이터베이스나 저장소를 나눠 관리하는 모든 기법을 뜻한다. 예를 들어 테이블을 날짜별로 나누는 **range partitioning**, 해시값 기준으로 나누는 **hash partitioning**이 있다.  
Sharding은 이를 더 분산 시스템 관점에서 확장한 형태로, 각 shard가 독립적인 저장소처럼 동작한다. 보통 **샤드 키**를 기준으로 어떤 데이터가 어느 노드에 갈지 결정한다. 예: `user_id % 4`로 4개 샤드에 분산.  
장점은 데이터가 많아져도 서버를 추가해 수평 확장이 가능하다는 점이다. 특정 shard에 장애가 나도 전체 시스템이 멈추지 않을 수 있다. 반면 **조인, 집계, 트랜잭션**이 shard를 넘나들면 복잡해진다. 샤드 키가 한쪽으로 치우치면 **hot shard**가 생겨 병목이 발생한다.  
실무에서는 조회 패턴, 데이터 증가 속도, 운영 복구 전략까지 같이 고려해야 한다.

## 짧은 예시/비유
도서관 책을 **장르별 선반**에 나누는 것은 partitioning이고, 각 장르를 **서로 다른 지점의 도서관**에 보내는 것은 sharding과 비슷하다.  
예를 들어 사용자 1억 명을 `user_id` 기준 8개 shard로 나누면, 특정 사용자의 읽기/쓰기를 해당 shard에서만 처리해 부하를 분산할 수 있다.

## 면접 질문
**Q1. Partitioning과 Sharding의 차이는?**  
A. Partitioning은 데이터를 나누는 일반 개념이고, Sharding은 그 조각을 여러 서버에 분산하는 분산 저장 방식이다.

**Q2. Sharding의 대표적인 단점은?**  
A. 샤드 간 조인/트랜잭션이 어려워지고, 샤드 키 설계가 나쁘면 데이터 쏠림과 재분배 비용이 커진다.

## 한 줄 정리
**Partitioning은 “나누기”, Sharding은 “나눠서 분산하기”이며, 확장성과 운영 복잡도의 트레이드오프를 이해하는 것이 핵심이다.**
