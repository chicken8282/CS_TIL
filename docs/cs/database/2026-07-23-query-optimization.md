# 쿼리 최적화: 자주 하는 실수와 안티패턴

- **측정 없이 최적화하지 말기:** `EXPLAIN`, 실행 시간, 읽은 행 수를 먼저 확인한다.
- **인덱스가 있어도 사용된다는 보장은 없음:** 함수, 암묵적 형 변환, 선행 와일드카드는 인덱스 활용을 막는다.
- **DB만 보지 말기:** N+1 조회, 과도한 데이터 전송, 잘못된 페이지네이션도 전체 성능을 악화시킨다.

## 개념 설명

쿼리 최적화는 같은 결과를 더 적은 CPU, 디스크 I/O, 메모리로 얻도록 실행 계획과 SQL을 개선하는 과정이다. 먼저 슬로 쿼리 로그나 APM으로 병목을 찾고, `EXPLAIN` 또는 `EXPLAIN ANALYZE`로 실제 실행 계획을 확인한다. 예상 행 수와 실제 행 수의 차이가 크면 통계 정보가 오래되었거나 조건 선택도가 잘못 추정된 것일 수 있다.

가장 흔한 안티패턴은 `SELECT *`이다. 사용하지 않는 컬럼까지 읽으면 디스크와 네트워크 비용이 늘고, 커버링 인덱스도 활용하기 어렵다. 필요한 컬럼만 명시하자. `WHERE DATE(created_at) = ...`처럼 인덱스 컬럼에 함수를 적용하는 것도 피한다. 범위 조건으로 바꾸면 인덱스를 탐색할 가능성이 높아진다.

`LIKE '%keyword'`는 선행 와일드카드 때문에 일반 B-Tree 인덱스를 거의 사용하지 못한다. 전문 검색이나 별도 검색 엔진을 고려한다. 또한 문자열과 숫자를 비교하는 암묵적 형 변환, `OR`가 과도하게 섞인 조건, 큰 `OFFSET` 기반 페이지네이션도 주의해야 한다. 후자는 뒤 페이지로 갈수록 많은 행을 건너뛰므로 커서 기반 페이지네이션이 적합하다.

인덱스를 무작정 추가하는 것도 실수다. 인덱스는 조회를 빠르게 하지만 `INSERT`, `UPDATE`, `DELETE` 비용과 저장 공간을 증가시킨다. 복합 인덱스는 보통 선택도가 높은 조건과 정렬·조인 조건을 함께 고려하며, 컬럼 순서가 중요하다. 단, DBMS와 데이터 분포에 따라 결과가 다르므로 반드시 실행 계획으로 검증한다.

애플리케이션에서는 ORM의 N+1 문제도 자주 발생한다. 목록을 한 번 조회한 뒤 각 항목마다 상세 쿼리를 실행하면 요청 수가 급증한다. 조인, 배치 조회, 적절한 eager loading으로 줄이되, 무분별한 조인은 중복 행과 큰 결과 집합을 만들 수 있으므로 함께 점검한다.

## SQL 예시

```sql
-- 안티패턴: 컬럼 함수, 전체 컬럼, 큰 OFFSET
SELECT *
FROM orders
WHERE DATE(created_at) = '2025-01-01'
ORDER BY id
LIMIT 50 OFFSET 100000;

-- 개선: 범위 검색, 필요한 컬럼, 커서 페이지네이션
SELECT id, user_id, total
FROM orders
WHERE created_at >= '2025-01-01'
  AND created_at <  '2025-01-02'
  AND id > :last_id
ORDER BY id
LIMIT 50;
```

## 인터뷰 질문

### 1. 인덱스가 있는데도 풀 스캔이 발생하는 이유는?

인덱스 컬럼에 함수를 적용했거나, 선행 와일드카드를 사용했거나, 암묵적 형 변환이 발생했을 수 있다. 또한 조건의 선택도가 낮아 옵티마이저가 풀 스캔을 더 저렴하다고 판단할 수도 있다.

### 2. `OFFSET` 페이지네이션의 문제와 대안은?

DB가 앞선 행을 읽고 버려야 하므로 뒤 페이지일수록 느려진다. 마지막으로 본 정렬 키를 조건에 사용하는 커서 기반 페이지네이션을 사용하면 일정한 성능을 기대할 수 있다.

> **한 줄 정리:** 실행 계획으로 병목을 측정한 뒤, 데이터 접근량과 쿼리 호출 수를 줄이는 방향으로 최적화하라.
