# HTTP 캐싱

- **목적**: 같은 리소스를 반복 요청할 때 서버 부하와 응답 지연을 줄인다.
- **핵심 방식**: 캐시가 `이 리소스를 다시 써도 되는가?`를 판단하는 정책과 검증 절차가 중요하다.
- **실무 포인트**: 정적 자원은 강하게 캐시하고, HTML/API 응답은 변경 가능성을 고려해 신중하게 설정한다.

## 개념 설명

HTTP 캐싱은 브라우저, 프록시, CDN이 응답을 저장해 두고 다음 요청에서 재사용하는 메커니즘이다.  
캐시는 크게 **신선도 판단**과 **재검증**으로 나뉜다.

- **신선도(Freshness)**: `Cache-Control: max-age=3600`처럼 정해진 시간 동안은 서버에 묻지 않고 바로 사용한다.
- **재검증(Validation)**: 캐시가 만료되었거나 불확실할 때 서버에 `If-None-Match`, `If-Modified-Since`를 보내 변경 여부를 확인한다.
- **ETag**: 리소스 버전 식별자. 변경되지 않았으면 서버는 `304 Not Modified`를 반환해 본문 전송을 생략한다.
- **Last-Modified**: 마지막 수정 시각 기반. ETag보다 정밀도가 낮아 보조적으로 사용된다.
- **Cache-Control**: `no-cache`는 저장은 가능하지만 매번 검증 필요, `no-store`는 저장 자체를 금지한다.

실무에서는 정적 파일에 파일명 해시를 붙여 `app.a1b2c3.js`처럼 배포하면, 새 버전과 구버전을 명확히 분리해 장기 캐시를 안전하게 적용할 수 있다.

## 짧은 예시 또는 비유

도서관 책 대출과 비슷하다.  
자주 보는 책을 집에 잠깐 보관해 두면 매번 도서관에 갈 필요가 없다. 다만 책이 개정되었는지 확인해야 하므로, 표지의 버전 번호(ETag)를 보고 같으면 그대로 읽고, 다르면 새 책으로 교체한다.

## 면접 질문 2개

1. **`max-age`와 `no-cache`의 차이는?**  
   `max-age`는 지정 시간 동안 바로 사용 가능하지만, `no-cache`는 저장은 가능해도 매 요청마다 서버 검증이 필요하다.

2. **304 Not Modified는 왜 중요한가?**  
   본문을 다시 내려주지 않아 대역폭과 응답 시간을 줄인다. 대규모 트래픽에서 효과가 크다.

## 한 줄 정리

HTTP 캐싱은 “자주 쓰는 응답을 재사용하되, 변경 여부는 정확히 검증하는 기술”이다.
