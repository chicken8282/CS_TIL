# Infrastructure as Code(IaC)

- **인프라는 코드로 관리**한다: 서버, 네트워크, DB, 권한, 배포 설정을 수동 클릭 대신 선언적으로 정의
- **재현성/일관성**이 핵심: 같은 코드면 같은 환경을 반복 생성 가능
- **면접 포인트**는 장점뿐 아니라 **드리프트, 상태 관리, 롤백, 보안**까지 설명하는 것

## 개념 설명

Infrastructure as Code(IaC)는 인프라를 사람이 직접 설정하는 대신 코드로 정의하고 자동으로 프로비저닝·변경·삭제하는 방식이다.  
예: Terraform, CloudFormation, Ansible, Pulumi.  
IaC의 핵심은 “현재 상태를 코드에 맞게 맞춘다”는 점이다. 이를 통해 개발/운영 환경 차이를 줄이고, 서버 수십 대를 동일하게 구성할 수 있다.

면접에서는 다음을 자주 묻는다:
- 왜 IaC가 필요한가?
- 수동 설정과 어떤 차이가 있는가?
- 상태(state)를 왜 관리해야 하는가?
- 배포 후 설정이 바뀌는 드리프트(drift)를 어떻게 막는가?

장점:
- 빠른 환경 복제와 확장
- 코드 리뷰, Git 히스토리로 변경 추적
- 반복 작업 자동화로 휴먼 에러 감소
- 롤백과 재현성 향상

주의점:
- 상태 파일 유출 시 보안 문제 가능
- 잘못된 변경이 대규모 장애로 이어질 수 있음
- 여러 사람이 동시에 변경하면 충돌 가능

## 짧은 예시 / 비유

비유하면, IaC는 **집을 짓는 설계도**다.  
수동 구성은 매번 현장에서 기억대로 짓는 것이고, IaC는 설계도대로 자재와 공정을 자동으로 맞추는 것이다.  
예를 들어 Terraform 코드에 `EC2 2대, 보안그룹 80/443 허용, RDS 1개`를 적으면, 실행 시 해당 인프라가 자동 생성된다.

## 면접 질문 2개

**Q1. IaC를 쓰면 무엇이 가장 좋아지나요?**  
A. 재현성과 일관성이 좋아진다. 환경을 코드로 관리하므로 테스트/스테이징/운영을 동일하게 구성하기 쉽다.

**Q2. IaC의 단점이나 주의점은?**  
A. 상태 관리와 보안이 중요하다. 상태 파일 노출, 드리프트, 잘못된 코드 변경이 큰 장애로 이어질 수 있다.

**한 줄 요약:** IaC는 인프라를 “수동 작업”이 아니라 “버전 관리되는 코드”로 다뤄 재현성과 운영 안정성을 높이는 방식이다.
