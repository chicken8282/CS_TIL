# CI/CD 기본기

- **CI(Continuous Integration)**: 코드 변경을 자주 통합하고 자동으로 빌드·테스트한다.
- **CD(Continuous Delivery/Deployment)**: 검증된 코드를 빠르게 배포 가능한 상태로 유지하거나, 자동 배포한다.
- **핵심 목적**: 배포 리스크를 줄이고, 피드백 속도를 높이며, 릴리스 품질을 안정화한다.

## 개념 설명

CI/CD는 개발자가 코드를 자주 올려도 시스템이 자동으로 검증하고 배포까지 이어지게 만드는 파이프라인이다.  
**CI**는 보통 `push`나 `pull request` 시점에 실행된다. 예를 들어 빌드, 단위 테스트, 정적 분석을 자동화해 “이 변경이 기존 코드와 충돌하지 않는지” 빠르게 확인한다. 핵심은 **작은 변경을 자주 통합**하는 것이다.

**CD**는 두 가지로 나뉜다.  
- **Continuous Delivery**: 배포 가능한 상태까지 자동화하되, 실제 배포는 사람이 승인할 수 있다.  
- **Continuous Deployment**: 검증을 통과하면 운영 환경에 자동 배포한다.

실무에서는 CI/CD 파이프라인에 보통 다음 단계가 포함된다.
1. 코드 체크아웃
2. 의존성 설치
3. 빌드
4. 테스트
5. 보안/품질 검사
6. 스테이징 배포
7. 운영 배포

중요한 점은 **자동화 자체가 목적이 아니라, 안전하고 빠른 릴리스를 위한 수단**이라는 것이다. 테스트가 부실하면 파이프라인은 빨라도 신뢰할 수 없다. 따라서 CI/CD는 테스트 전략, 브랜치 전략, 롤백 전략과 함께 설계해야 한다.

## 짧은 예시 / 비유

비유하자면 CI/CD는 **자동화된 품질 검사 라인을 갖춘 택배 시스템**과 같다.  
상품(코드)을 매번 조금씩 보내면, 검사대(테스트)를 통과한 것만 다음 단계로 넘어가고, 최종 목적지(운영)에 빠르게 도착한다.  
예: `main` 브랜치에 머지되면 GitHub Actions가 테스트를 돌리고, 성공하면 스테이징에 배포한 뒤 승인 후 운영 배포를 수행한다.

## 면접 질문 2개

**Q1. CI와 CD의 차이는 무엇인가요?**  
A1. CI는 코드 통합 시 자동 빌드/테스트로 품질을 확인하는 것이고, CD는 검증된 코드를 배포 가능한 상태로 유지하거나 실제로 자동 배포하는 것이다.

**Q2. CI/CD 도입 시 가장 중요한 고려사항은?**  
A2. 테스트 신뢰도와 롤백 가능성이다. 자동 배포가 빨라도 테스트가 부실하면 위험하고, 장애 시 빠르게 되돌릴 수 있어야 한다.

**한 줄 요약:** CI/CD는 “자주 통합하고, 자동 검증하며, 안전하게 빠르게 배포하는” 개발 운영 방식이다.
