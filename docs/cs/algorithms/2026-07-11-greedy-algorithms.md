# 그리디 알고리즘(Greedy Algorithm)

- **매 순간 최선의 선택**을 하며 전체 최적해를 노리는 방법이다.
- 구현은 단순하지만, **정당성 증명**이 어려워 실무에서는 검증이 중요하다.
- 최적해가 보장되지 않는 문제에 잘못 적용하면 **성능보다 더 큰 버그**를 만든다.

## Concept explanation

그리디 알고리즘은 현재 시점에서 가장 좋아 보이는 선택을 반복하는 전략이다.  
예: 작업 스케줄링에서 “가장 빨리 끝나는 작업부터 처리”하면 처리량을 높일 수 있다.  
하지만 모든 문제에 통하는 것은 아니다. 핵심은 **지역 최적 선택이 전역 최적해로 이어지는 구조**인지 확인하는 것이다.

실무 트러블슈팅 관점에서는 다음을 체크한다.

- **반례 탐색**: 입력 경계값, 중복, 0/음수, 큰 수에서 선택이 깨지지 않는지 확인
- **정렬 기준 검증**: 코드상 `sort` 한 번으로 끝나도, 기준이 문제 조건과 정확히 일치하는지 확인
- **증명 로그 확보**: 왜 이 선택이 맞는지 팀 리뷰나 장애 대응 시 설명 가능해야 함

대표 조건은 두 가지다:
1. **탐욕 선택 속성**: 한 번의 선택이 이후 최적성에 영향을 주지 않음  
2. **최적 부분 구조**: 부분 문제의 최적해가 전체 최적해 구성에 사용됨

## Short example or analogy

예를 들어 회의실 예약에서 “가장 빨리 끝나는 회의부터 선택”하면 하루에 배치 가능한 회의 수가 늘어난다.  
반대로 “가장 긴 회의부터” 넣으면 일정이 쉽게 꼬인다.  
실무에서도 캐시 eviction, 우선순위 큐 기반 작업 처리처럼 **짧은 선택 규칙이 전체 효율을 좌우**하는 경우가 많다.

## Interview questions

**Q1. 그리디와 DP의 차이는?**  
A1. 그리디는 매 단계 최선 선택을 고정하고, DP는 모든 상태를 비교해 최적해를 찾는다.

**Q2. 그리디가 맞는지 어떻게 검증하나?**  
A2. 탐욕 선택이 최적해를 해치지 않는다는 증명과, 반례 테스트를 통해 확인한다.

## One-line takeaway

**그리디는 “빠른 최선”이 아니라 “증명 가능한 최선”일 때만 안전하다.**
