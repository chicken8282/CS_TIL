# 메모리 매핑 파일(Memory-Mapped File)

- 파일을 프로세스의 가상 주소 공간에 매핑해 일반 메모리처럼 접근한다.
- 반복적인 `read`/`write`와 사용자 공간·커널 공간 간 복사를 줄여 대용량·랜덤 접근에 유리하다.
- 페이지 폴트, 동기화, 파일 크기 변경을 이해하지 않으면 성능 저하와 데이터 손상이 발생한다.

## 개념 설명

메모리 매핑 파일은 파일의 일부 또는 전체를 프로세스의 가상 메모리 주소에 연결하는 기능이다. 애플리케이션이 매핑된 주소를 읽거나 쓰면 운영체제가 필요한 파일 페이지를 페이지 캐시에 적재한다. 따라서 파일 접근이 메모리 접근처럼 보이지만, 실제로는 페이지 폴트와 디스크 I/O가 발생할 수 있다.

일반적인 `read` 방식은 커널 버퍼에서 사용자 버퍼로 데이터를 복사한다. 반면 매핑 방식은 페이지 캐시를 프로세스 주소 공간에서 직접 바라보므로 큰 파일의 랜덤 접근, 여러 프로세스 간 공유에 적합하다. 단, “매핑하면 파일 전체가 RAM에 올라간다”는 오해가 대표적인 실수다. 실제 메모리 사용량은 접근한 페이지와 운영체제의 캐시 정책에 좌우된다.

자주 하는 실수는 다음과 같다.

- **파일 크기를 먼저 확보하지 않음**: 매핑 범위를 파일보다 크게 잡고 접근하면 플랫폼에 따라 예외나 `SIGBUS`가 발생할 수 있다.
- **`flush`를 영속성 보장으로 오해함**: `flush`는 변경 내용을 커널 또는 저장장치로 밀어내는 요청이지, 모든 하드웨어 장애까지 막는 트랜잭션이 아니다. 강한 내구성이 필요하면 파일 동기화 정책을 별도로 검토한다.
- **동시 접근을 일반 메모리처럼 처리함**: 여러 프로세스가 같은 영역을 수정하면 경쟁 조건과 찢어진 레코드가 생긴다. 락, 원자적 프로토콜, append 설계가 필요하다.
- **매핑 수명과 파일 수명을 무시함**: 파일을 닫거나 크기를 줄인 뒤 매핑을 계속 사용하지 않는다.
- **작은 파일·순차 읽기에 무조건 사용함**: 매핑 생성 비용, 페이지 폴트, 주소 공간 부담 때문에 단순 버퍼 I/O보다 느릴 수 있다.
- **레코드 경계를 고려하지 않음**: 고정 길이 구조체는 정렬·엔디언·버전 문제를 만들 수 있다. 파일 포맷을 명시적으로 설계한다.

특히 매핑된 데이터를 포인터처럼 오래 보관하는 안티패턴을 피한다. 매핑 해제 후 해당 참조를 사용하면 오류가 발생한다. 또한 파일을 외부 프로세스가 truncate할 수 있는 환경에서는 파일 잠금과 크기 변경 정책을 함께 둬야 한다.

```python
import mmap

with open("data.bin", "r+b") as f:
    size = f.seek(0, 2)
    if size == 0:
        raise ValueError("empty file")
    f.seek(0)
    with mmap.mmap(f.fileno(), size, access=mmap.ACCESS_WRITE) as mm:
        old = mm[0:4]
        mm[0:4] = b"CS!!"
        mm.flush()
```

## 면접 질문

1. **메모리 매핑 파일이 `read`보다 항상 빠른가?**  
   아니다. 랜덤 접근과 대용량 공유에는 유리하지만, 작은 파일이나 순차 접근에서는 매핑 비용과 페이지 폴트 때문에 일반 I/O가 더 빠를 수 있다.

2. **매핑 영역을 여러 프로세스가 동시에 수정하면 어떻게 해야 하는가?**  
   매핑 자체는 동기화를 제공하지 않는다. 프로세스 간 락, 원자적 자료구조, 단일 작성자 모델 등을 사용해 경쟁 조건과 부분 갱신을 막아야 한다.

> **한 줄 요약:** 메모리 매핑 파일은 파일을 메모리처럼 다루는 도구이지, 동기화·내구성·성능을 자동으로 보장하는 기능은 아니다.
