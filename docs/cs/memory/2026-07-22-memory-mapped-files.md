# 메모리 매핑 파일과 성능 최적화

- **파일을 프로세스의 가상 주소 공간에 매핑**하여 일반적인 `read/write` 호출 없이 메모리 접근처럼 다룬다.
- 성능은 **페이지 폴트, 메모리 압력, 접근 패턴, 파일 크기**에 크게 좌우된다.
- 복사 감소와 랜덤 접근에는 유리하지만, 항상 빠른 것은 아니며 **순차 접근·작은 파일·캐시 상태**를 함께 고려해야 한다.

## 개념 설명

메모리 매핑 파일은 운영체제가 파일의 일부 또는 전체를 프로세스의 가상 메모리 주소에 연결하는 기능이다. 애플리케이션이 특정 주소를 읽으면 해당 페이지가 메모리에 없을 때 페이지 폴트가 발생하고, 운영체제가 디스크에서 필요한 페이지를 페이지 캐시에 적재한다. 이후 같은 페이지를 읽으면 시스템 콜과 사용자 버퍼 복사를 줄일 수 있다.

일반적인 `read()` 방식은 커널 버퍼에서 사용자 버퍼로 데이터 복사가 발생한다. 반면 `mmap()`은 페이지 캐시를 프로세스 주소 공간에서 직접 참조하므로 대용량 파일의 랜덤 접근, 인덱스 파일, 데이터베이스의 파일 기반 구조에 적합하다. 다만 최초 접근 시 지연이 발생할 수 있고, 페이지 단위로 로드되므로 실제로 사용하지 않는 영역까지 매핑하면 메모리 압박이 커진다.

성능 최적화에서는 접근 패턴이 핵심이다. 순차 스캔이라면 운영체제에 순차 접근 힌트를 제공하고, 랜덤 접근이라면 필요한 구간만 매핑하거나 윈도우 방식으로 나눈다. 파일 전체를 한 번에 매핑하는 방식은 주소 공간과 페이지 캐시를 불필요하게 점유할 수 있다. 자주 읽는 데이터는 매핑만으로 충분하지 않을 수 있으므로 핫 데이터 구조를 작게 유지하고, 필요하면 사전 프리페치 또는 `madvise()`를 사용한다.

쓰기 시에는 변경 내용이 즉시 디스크에 기록된다고 가정하면 안 된다. `flush` 또는 동기화 정책이 필요하며, 장애 시 일관성을 위해 원자적 갱신과 로그 구조를 고려해야 한다. 여러 프로세스가 같은 매핑을 수정하면 일반 메모리와 동일하게 경쟁 조건이 발생하므로 락이나 원자적 연산이 필요하다. 32비트 프로세스에서는 큰 파일 매핑이 주소 공간 부족으로 실패할 수 있다.

```python
import mmap

with open("records.bin", "r+b") as f:
    mm = mmap.mmap(f.fileno(), 0, access=mmap.ACCESS_WRITE)
    for offset in range(0, len(mm), 4096):
        value = mm[offset:offset + 8]
        if value == b"TARGET!!":
            mm[offset:offset + 8] = b"FOUND!!!"
    mm.flush()
    mm.close()
```

## 면접 질문

### 1. `mmap()`이 `read()`보다 항상 빠른가?

아니다. `mmap()`은 사용자 버퍼 복사와 시스템 콜을 줄일 수 있지만, 최초 페이지 폴트 비용과 페이지 관리 비용이 있다. 작은 파일의 단순 순차 읽기나 이미 캐시된 데이터를 대량으로 처리하는 경우에는 `read()`가 더 빠를 수도 있다.

### 2. 메모리 매핑 파일의 성능을 어떻게 개선할 수 있는가?

접근 패턴에 맞게 매핑 범위를 제한하고, 순차·랜덤 접근 힌트를 제공한다. 페이지 폴트를 측정하며 프리페치, 적절한 버퍼 크기, 파일 배치 개선을 적용하고, 불필요한 쓰기와 잦은 `flush`를 줄인다.

> **한 줄 요약:** 메모리 매핑 파일은 복사와 시스템 콜을 줄여 랜덤 접근을 최적화하지만, 페이지 폴트와 캐시·동기화 비용까지 측정해야 제대로 빠르다.
