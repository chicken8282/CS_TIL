# OAuth 2.0

- **사용자 비밀번호를 앱에 직접 넘기지 않고**, 권한만 위임하는 표준 프로토콜이다.
- 핵심 주체는 **Resource Owner(사용자), Client(앱), Authorization Server(인증 서버), Resource Server(API 서버)** 이다.
- 실무에서 **소셜 로그인, 외부 API 연동, 토큰 기반 인증/인가**에 자주 쓰인다.

## 개념 설명

OAuth 2.0은 **인증(Authentication)** 이 아니라 **인가(Authorization)** 를 위한 기술이다.  
즉, “당신이 누구인지”를 확인하기보다 “이 앱이 사용자의 데이터를 어디까지 사용할 수 있는지”를 결정한다.

흐름은 보통 다음과 같다.

1. 사용자가 앱에서 “구글로 로그인” 같은 버튼을 누른다.
2. 앱은 사용자를 Authorization Server로 보낸다.
3. 사용자는 자신의 계정으로 로그인하고, 앱에 권한을 허용한다.
4. Authorization Server는 **Access Token** 을 앱에 준다.
5. 앱은 이 토큰으로 Resource Server에 API 요청을 보낸다.

중요한 점:
- **Access Token**: 실제 API 호출에 사용
- **Refresh Token**: Access Token 만료 시 새 토큰 발급에 사용
- 토큰이 있더라도 **범위(scope)** 가 제한되므로 필요한 권한만 부여 가능

면접에서는 “OAuth는 로그인 자체가 아니라 권한 위임”이라는 점을 정확히 말하는 것이 중요하다.

## 짧은 예시 / 비유

회사 출입을 생각하면 쉽다.  
비밀번호를 동료에게 주는 대신, **“회의실만 들어갈 수 있는 출입증”** 을 발급해 주는 것이다.  
출입증이 Access Token이고, 입장 가능한 구역이 Scope다.

예: 사진 앱이 구글 드라이브의 “사진 폴더만 읽기” 권한을 받는다면, 전체 드라이브가 아니라 일부만 접근 가능하다.

## 면접 질문 2개

**Q1. OAuth 2.0과 로그인은 같은 개념인가요?**  
A1. 아니다. OAuth는 권한 위임이고, 로그인은 인증이다. 로그인은 보통 OpenID Connect를 함께 쓴다.

**Q2. Access Token이 유출되면 왜 위험한가요?**  
A2. 토큰이 곧 API 접근 권한이기 때문이다. 유출되면 만료 전까지 권한 범위 내에서 악용될 수 있다.

## 한 줄 요약

**OAuth 2.0은 비밀번호를 공유하지 않고, 토큰으로 제한된 권한만 안전하게 위임하는 표준이다.**
