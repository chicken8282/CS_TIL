# OAuth 2.0 핵심 정리

- **사용자 비밀번호를 앱에 직접 주지 않고** 다른 서비스의 자원 접근을 허용하는 **인가(Authorization) 프레임워크**다.
- 핵심 주체는 **Resource Owner, Client, Authorization Server, Resource Server**이며, **Access Token**으로 접근한다.
- 로그인과 자주 함께 쓰이지만, **OAuth 2.0 자체는 인증(Authentication)이 아니라 인가**를 다룬다.

## 개념 설명

OAuth 2.0은 사용자가 특정 앱에게 “내 계정 정보에 접근해도 된다”는 권한을 위임하는 표준이다.  
예를 들어 사진 편집 앱이 구글 드라이브의 사진만 읽고 싶을 때, 사용자는 구글 로그인 화면에서 권한을 승인한다. 앱은 사용자의 비밀번호를 받지 않고 **액세스 토큰**을 받아 API를 호출한다.

흐름은 보통 다음과 같다.
1. 사용자가 클라이언트 앱에서 로그인/연결 요청
2. 앱이 Authorization Server로 이동시킴
3. 사용자가 권한 동의
4. Authorization Server가 **Authorization Code** 발급
5. 클라이언트가 코드를 토큰으로 교환
6. 토큰으로 Resource Server API 접근

실무에서는 **Authorization Code Grant + PKCE**가 모바일/웹에서 많이 쓰인다.  
중요 포인트는 **토큰 유출 방지**, **스코프(scope) 최소화**, **짧은 만료 시간**, **Refresh Token 관리**다.

## 짧은 예시 / 비유

호텔 비유로 보면,  
- **비밀번호**는 호텔 본인 키  
- **Access Token**은 특정 시간만 유효한 객실 출입 카드  
- **Scope**는 “수영장만 사용 가능” 같은 제한 권한이다.  

즉, 다른 사람에게 집 열쇠를 주는 대신, 필요한 공간만 열리는 임시 카드를 주는 방식이다.

## 면접 질문 2개

**Q1. OAuth 2.0과 로그인(Authentication)은 같은가요?**  
A. 아니다. OAuth 2.0은 자원 접근 권한을 주는 인가 규격이고, 로그인은 사용자를 식별하는 인증이다. 다만 OpenID Connect가 인증을 보완한다.

**Q2. Authorization Code Flow를 쓰는 이유는?**  
A. 액세스 토큰을 브라우저에 직접 노출하지 않고, 서버 간 교환으로 안전성을 높이기 때문이다. PKCE를 쓰면 모바일/SPA에서도 더 안전하다.

## 한 줄 요약

**OAuth 2.0은 비밀번호를 넘기지 않고, 토큰으로 필요한 권한만 위임하는 인가 표준이다.**
