# OAuth 2.0

- **권한 부여(Authorization) 표준**이다. 사용자가 비밀번호를 직접 넘기지 않고, 서비스 간에 제한된 접근 권한을 위임한다.
- **인증(Authentication)과 다르다.** OAuth 2.0은 “누구인지 확인”보다 “무엇을 할 수 있는지 허용”에 초점이 있다.
- **Access Token이 핵심**이다. 짧은 수명의 토큰으로 API 접근을 허용하고, 필요 시 Refresh Token으로 재발급한다.

## 개념 설명
OAuth 2.0은 리소스 소유자(사용자), 클라이언트(앱), 권한 서버(Authorization Server), 리소스 서버(API)로 구성된다.  
예를 들어 사용자가 “사진 앱”에 “구글 드라이브의 사진만 읽기” 권한을 줄 수 있다. 이때 앱은 사용자의 구글 비밀번호를 받지 않고, 구글이 발급한 Access Token으로 API를 호출한다.  
대표 흐름은 Authorization Code Grant이다. 사용자는 구글 로그인 페이지로 이동해 동의하고, 서버는 **인가 코드(code)** 를 받는다. 앱 서버는 이 코드를 권한 서버에 보내 Access Token으로 교환한다. 이 방식은 클라이언트에 토큰을 바로 노출하지 않아 안전하다.  
실무에서 중요한 점은 **스코프(scope)** 로 권한을 최소화하고, Access Token은 짧게, Refresh Token은 더 엄격하게 보호하는 것이다.

## 짧은 예시 / 비유
호텔 키카드와 비슷하다. 프런트에서 신분 확인 후 방 번호에 맞는 키카드를 발급받고, 그 카드로 엘리베이터와 객실만 접근한다. 호텔의 메인 금고 비밀번호를 카드에 적어 두지 않는 것과 같다.

## 면접 질문
**Q1. OAuth 2.0과 로그인은 같은가요?**  
A. 아니다. OAuth는 권한 위임 표준이고, 로그인은 인증 문제다. 다만 OpenID Connect를 함께 쓰면 로그인 기능을 구현한다.

**Q2. 왜 Authorization Code Flow를 많이 쓰나요?**  
A. Access Token을 브라우저에 직접 노출하지 않고 서버 간 교환으로 처리해 보안성이 높기 때문이다.

**한 줄 요약:** OAuth 2.0은 비밀번호를 공유하지 않고, 제한된 토큰으로 API 접근 권한을 안전하게 위임하는 표준이다.
