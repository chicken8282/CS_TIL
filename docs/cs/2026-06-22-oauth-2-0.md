# OAuth 2.0

- **인증(Authentication)**이 아니라 **인가(Authorization)**를 위한 표준 프로토콜
- 사용자가 비밀번호를 직접 넘기지 않고, **Access Token**으로 권한을 위임
- **Refresh Token**과 **Scope**를 이해하면 실무 흐름이 보인다

## Concept explanation

OAuth 2.0은 제3자 앱이 사용자의 자원에 접근할 수 있도록 **권한을 위임**하는 방식이다. 예를 들어 “구글 계정으로 로그인”은 흔히 인증처럼 보이지만, OAuth 2.0 자체는 “이 앱이 내 구글 캘린더에 접근해도 되는가?”를 다룬다.  
핵심 등장인물은 다음과 같다.

- **Resource Owner**: 사용자
- **Client**: 권한을 요청하는 앱
- **Authorization Server**: 로그인/동의 후 토큰 발급
- **Resource Server**: 실제 API 제공 서버

흐름은 보통 이렇다. 사용자가 클라이언트에서 로그인 버튼을 누르면, 인증 서버로 이동해 동의 화면을 본다. 사용자가 승인하면 **Authorization Code**를 받고, 클라이언트는 이를 서버 측에서 교환해 **Access Token**을 얻는다. 이 토큰으로 API를 호출한다. Access Token은 짧게, Refresh Token은 길게 두어 보안을 높인다.  
실무에서는 **PKCE**가 중요한데, 모바일/SPA처럼 클라이언트 비밀키를 안전하게 보관하기 어려운 환경에서 코드 탈취 위험을 줄인다.

## Short example or analogy

호텔 키카드에 비유할 수 있다.  
- 호텔 프런트 = Authorization Server  
- 키카드 = Access Token  
- 객실 출입 권한 = Scope  
- 체크인 후 잠깐 쓰는 카드라 분실해도 피해를 줄임

즉, 비밀번호를 계속 보여주는 대신, 필요한 권한만 담은 임시 카드만 발급받는 구조다.

## Interview questions

**Q1. OAuth 2.0은 인증인가, 인가인가?**  
A. 인가다. 사용자가 누구인지 확인하는 인증은 별개이며, OAuth는 자원 접근 권한 위임에 초점이 있다.

**Q2. Authorization Code Grant를 쓰는 이유는?**  
A. Access Token을 직접 노출하지 않고 서버에서 교환해 보안을 높이기 위해서다. PKCE를 함께 쓰면 더 안전하다.

## One-line takeaway

OAuth 2.0은 “비밀번호 공유” 대신 “토큰 기반 권한 위임”으로 안전하게 외부 앱 접근을 허용하는 표준이다.
