# OAuth 2.0

- **권한 부여 프레임워크**다: 사용자가 비밀번호를 직접 넘기지 않고, 앱이 제한된 자원 접근 권한만 얻는다.
- **Access Token 중심**으로 동작한다: 짧은 수명의 토큰으로 API를 호출하고, 필요 시 Refresh Token으로 갱신한다.
- **인증(Authentication)과 구분**해야 한다: OAuth 2.0은 “누구인가”보다 “무엇을 허용할 것인가”에 초점이 있다.

## Concept explanation

OAuth 2.0은 제3자 애플리케이션이 사용자의 자원에 접근하도록 **위임된 권한**을 받는 표준이다. 예를 들어, 사진 편집 앱이 구글 포토에 접근해야 할 때 사용자는 구글 계정 비밀번호를 앱에 주지 않고, 구글의 동의 화면에서 필요한 권한만 승인한다. 이후 앱은 **Authorization Server**로부터 Access Token을 받아 **Resource Server(API)** 에 요청한다.

핵심 구성요소는 다음과 같다.
- **Resource Owner**: 자원의 소유자(사용자)
- **Client**: 권한을 요청하는 앱
- **Authorization Server**: 사용자 동의 후 토큰 발급
- **Resource Server**: 토큰을 검증하고 자원 제공

대표 흐름은 **Authorization Code Flow**다. 사용자는 로그인 및 동의 후 authorization code를 받고, 클라이언트는 이를 서버에서 교환해 access token을 받는다. 이 방식은 토큰 노출 위험을 줄여 모바일/웹 서비스에서 가장 널리 쓰인다. 인터뷰에서는 “OAuth는 인증이 아니라 인가”라는 점과, “토큰은 짧게, 갱신은 안전하게”가 자주 나온다.

## Short example or analogy

호텔 키카드로 비유하면, 비밀번호는 프런트에 맡기지 않는다. 대신 특정 시간 동안 특정 층만 열 수 있는 키카드를 받는다. 키카드가 **Access Token**, 갱신 데스크가 **Refresh Token**에 가깝다.

## Interview questions

1. **OAuth 2.0과 로그인은 같은가?**  
   아니요. OAuth 2.0은 자원 접근 권한을 위임하는 표준이고, 로그인은 사용자를 식별하는 인증이다.

2. **Refresh Token은 왜 필요한가?**  
   Access Token을 짧게 유지해 보안을 높이면서도, 사용자가 자주 재로그인하지 않도록 새 토큰을 발급받기 위해서다.

**Takeaway:** OAuth 2.0은 “비밀번호 공유 없이, 필요한 권한만 안전하게 위임하는 방식”이다.
