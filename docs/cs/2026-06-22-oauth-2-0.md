# OAuth 2.0

- **제3자 앱에 비밀번호를 주지 않고도 권한만 위임**하는 표준 인증/인가 프레임워크다.
- 핵심은 **Resource Owner, Client, Authorization Server, Resource Server**의 역할 분리다.
- 실무에서는 **Authorization Code + PKCE**가 가장 중요하며, 보안상 가장 많이 쓰인다.

## 개념 설명

OAuth 2.0은 사용자가 자신의 계정 자원에 대해 **다른 서비스가 제한된 권한으로 접근하도록 허용**하는 방식이다.  
예를 들어 “구글 계정으로 로그인”은 흔히 인증처럼 보이지만, OAuth 2.0 자체는 본질적으로 **인가(authorization)** 에 가깝다.  
즉, “이 앱에게 내 연락처 읽기 권한을 줄 것인가?”를 다루며, 사용자의 비밀번호는 클라이언트 앱이 알 필요가 없다.

흐름은 보통 다음과 같다.

1. 사용자가 클라이언트 앱에서 “구글로 로그인” 클릭
2. 클라이언트가 인증 서버로 리다이렉트
3. 사용자가 로그인 후 권한 동의
4. 인증 서버가 **authorization code** 발급
5. 클라이언트가 code를 서버에 보내 **access token** 교환
6. 클라이언트가 access token으로 리소스 서버 API 호출

여기서 **access token**은 짧게 쓰는 토큰이고, **refresh token**은 access token이 만료됐을 때 재발급에 사용된다.  
보안상 토큰은 노출되면 위험하므로, 프론트엔드 단독 앱에서는 **PKCE**를 사용해 코드 탈취 공격을 막는다.

## 짧은 예시 / 비유

OAuth 2.0은 **호텔 키카드**와 비슷하다.  
호텔 비밀번호를 직접 알려주는 대신, 특정 층과 방만 열리는 키카드를 받는 것처럼, 앱은 비밀번호 대신 **제한된 토큰**만 받는다.  
키카드가 만료되면 프론트데스크에서 다시 발급받듯이, access token도 refresh token으로 갱신할 수 있다.

## 면접 질문 2개

**Q1. OAuth 2.0과 로그인은 같은가요?**  
A. 아니요. OAuth 2.0은 본질적으로 인가입니다. 로그인은 보통 OpenID Connect(OIDC) 같은 인증 계층이 추가된 경우가 많습니다.

**Q2. Authorization Code Grant를 왜 많이 쓰나요?**  
A. access token을 브라우저에 직접 노출하지 않아 비교적 안전하고, 서버 간 토큰 교환 구조로 보안성이 높기 때문입니다.

## 한 줄 요약

**OAuth 2.0은 비밀번호를 공유하지 않고, 제한된 토큰으로 제3자 앱에 권한을 안전하게 위임하는 표준이다.**
