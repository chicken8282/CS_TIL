# OAuth 2.0 핵심 정리

- **인가(Authorization) 프레임워크**: 사용자가 비밀번호를 직접 넘기지 않고, 앱이 제한된 권한만 위임받게 한다.
- **토큰 기반**: Access Token으로 자원 서버에 접근하고, 만료 시 Refresh Token으로 재발급받는다.
- **역할 분리**: Resource Owner(사용자), Client(앱), Authorization Server(인증/인가 서버), Resource Server(API)로 나뉜다.

## 개념 설명
OAuth 2.0은 **“누가 로그인했는가”를 증명하는 인증(Authentication)**이 아니라, **“어떤 권한을 허용할 것인가”를 관리하는 인가(Authorization)** 표준이다. 예를 들어, 외부 사진 편집 앱이 내 구글 드라이브의 사진만 읽도록 허용할 수 있다. 이때 앱은 내 계정을 직접 알 필요가 없고, 구글이 발급한 토큰만 사용한다.

대표 흐름은 다음과 같다.  
1. 사용자가 Client에서 “구글 드라이브 접근 허용”을 누른다.  
2. Authorization Server가 사용자를 로그인시키고 동의 화면을 보여준다.  
3. 동의 후 Authorization Code를 발급한다.  
4. Client는 이 코드를 서버로 보내 Access Token을 받는다.  
5. Client는 Access Token으로 Resource Server API를 호출한다.

실무에서 중요한 점:
- **Access Token은 짧게**, **Refresh Token은 더 엄격하게 보호**
- 토큰은 URL 쿼리보다 **Authorization Header**로 전달
- **scope**로 권한 범위를 최소화해야 함
- SPA/모바일은 보안상 **Authorization Code + PKCE**가 일반적

## 짧은 예시/비유
호텔 키카드처럼 생각하면 쉽다. 프런트 데스크가 Authorization Server, 키카드가 Access Token, 객실 문이 Resource Server다. 투숙객은 매번 주민등록증을 보여주지 않고, 키카드로 제한된 기간만 출입한다.

## 면접 질문
**Q1. OAuth 2.0과 인증은 같은가요?**  
A. 아니다. OAuth 2.0은 인가를 위한 표준이고, 로그인 인증은 보통 OpenID Connect가 담당한다.

**Q2. Access Token과 Refresh Token의 차이는?**  
A. Access Token은 API 접근용, 짧게 만료된다. Refresh Token은 새 Access Token을 받는 데 쓰이며 더 안전하게 보관해야 한다.

## 한 줄 요약
OAuth 2.0은 비밀번호 공유 없이, 토큰으로 **권한만 안전하게 위임**하는 표준이다.
