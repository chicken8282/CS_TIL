# OAuth 2.0 핵심 정리

- **OAuth 2.0은 인증(Authentication)이 아니라 권한 부여(Authorization) 표준**이다.
- **사용자 비밀번호를 외부 서비스에 직접 주지 않고**, 토큰으로 제한된 접근 권한을 위임한다.
- **Access Token, Refresh Token, Scope**가 핵심이며, 모바일/웹/서버 API 연동에 자주 쓰인다.

## 개념 설명
OAuth 2.0은 한 서비스가 다른 서비스의 자원에 접근할 수 있도록 **사용자 동의 기반의 권한 위임**을 제공한다.  
예를 들어, 어떤 앱이 구글 캘린더에 접근하려면 사용자는 구글 로그인 화면에서 “이 앱이 캘린더를 읽을 수 있음”을 허용한다. 이때 앱은 비밀번호를 받지 않고 **Access Token**을 받아 API를 호출한다.

핵심 요소는 다음과 같다.
- **Resource Owner**: 사용자
- **Client**: 외부 앱
- **Authorization Server**: 토큰 발급 서버
- **Resource Server**: 실제 API 서버

**Access Token**은 짧게 쓰는 인증 수단이고, **Refresh Token**은 Access Token이 만료됐을 때 새 토큰을 발급받기 위한 수단이다.  
또한 **Scope**로 접근 범위를 제한한다. 예: `read_profile`, `read_calendar`.

실무에서 가장 많이 보는 흐름은 **Authorization Code Grant**다. 사용자가 브라우저에서 로그인하고, 서버가 코드와 토큰을 교환하는 방식이라 비교적 안전하다. SPA나 모바일 앱에서는 **PKCE**를 함께 사용해 코드 탈취 위험을 줄인다.

## 짧은 예시/비유
OAuth 2.0은 **호텔 카드키**와 비슷하다.  
프런트 데스크가 방 번호에 맞는 카드키를 발급하면, 투숙객은 방을 열 수 있지만 호텔 전체 열쇠를 받는 것은 아니다. 카드키가 토큰, 접근 가능한 방 범위가 Scope라고 보면 된다.

## 면접 질문
**Q1. OAuth 2.0과 인증은 같은가요?**  
A1. 아니다. OAuth 2.0은 사용자를 확인하는 인증이 아니라, 이미 인증된 사용자의 자원 접근 권한을 위임하는 표준이다.

**Q2. Access Token과 Refresh Token의 차이는?**  
A2. Access Token은 API 호출용으로 짧게 사용하고, Refresh Token은 만료된 Access Token을 재발급받는 데 사용한다. 보통 Refresh Token이 더 민감하게 관리된다.

## 한 줄 요약
**OAuth 2.0은 비밀번호 공유 없이 토큰으로 안전하게 권한을 위임하는 표준이다.**
