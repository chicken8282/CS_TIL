# OAuth 2.0: 안전한 권한 위임 표준

- **OAuth 2.0은 인증이 아니라 권한 부여(Authorization) 표준**이다.
- **사용자 비밀번호를 앱에 직접 주지 않고, 액세스 토큰으로 API 접근을 허용**한다.
- **Authorization Code, Refresh Token, Scope를 이해하면 실무와 면접에 강해진다.**

## 개념 설명
OAuth 2.0은 제3자 앱이 사용자를 대신해 특정 자원에 접근하도록 **권한을 위임**하는 프로토콜이다. 예를 들어, 사진 편집 앱이 사용자의 구글 드라이브 사진만 읽게 하고 싶을 때, 앱은 사용자의 구글 비밀번호를 받지 않는다. 대신 권한 서버가 발급한 **액세스 토큰**을 받아 API를 호출한다.

흐름은 보통 다음과 같다.  
1. 사용자가 서비스에 로그인 요청  
2. 앱이 권한 서버로 리다이렉트  
3. 사용자가 동의  
4. 권한 서버가 **Authorization Code** 발급  
5. 앱이 코드를 서버에서 검증 후 **Access Token** 획득  
6. 토큰으로 리소스 서버 API 접근

중요 개념:
- **Access Token**: 짧은 수명의 접근 증명서
- **Refresh Token**: 만료된 토큰을 재발급받는 데 사용
- **Scope**: 접근 가능한 범위 최소화
- **Redirect URI**: 토큰 탈취 방지를 위한 검증 대상

주의할 점은 OAuth 2.0은 “누구인가”를 검증하는 **인증(Authentication)** 자체가 아니라는 것이다. 로그인 기능이 필요하면 보통 **OpenID Connect(OIDC)** 를 함께 사용한다.

## 짧은 예시/비유
OAuth 2.0은 **호텔 키카드**와 비슷하다.  
프런트 데스크가 비밀번호를 주는 대신, “3층 객실만 출입 가능”한 키카드를 발급한다. 키카드는 특정 권한만 담고 있고, 만료되면 다시 발급받아야 한다.

## 면접 질문 2개
**Q1. OAuth 2.0과 인증의 차이는?**  
A. OAuth 2.0은 리소스 접근 권한 위임이고, 인증은 사용자가 누구인지 확인하는 과정이다.

**Q2. Authorization Code Flow를 쓰는 이유는?**  
A. 토큰을 브라우저에 직접 노출하지 않고, 서버 간 교환으로 보안을 높이기 위해서다.

**한 줄 정리:** OAuth 2.0은 비밀번호 공유 없이 토큰으로 필요한 권한만 안전하게 위임하는 표준이다.
