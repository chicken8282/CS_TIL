# Cache Invalidation

- **캐시는 빨라서 좋지만, 오래되면 틀릴 수 있다.**
- **무효화 전략이 성능과 정확도의 균형을 결정한다.**
- **정답은 하나가 아니라, 데이터 특성에 맞는 방식 선택이다.**

## 개념 설명

Cache Invalidation(캐시 무효화)은 **캐시에 저장된 값이 더 이상 유효하지 않을 때 이를 삭제하거나 갱신하는 과정**이다.  
캐시는 읽기 성능을 크게 높이지만, 원본 데이터(DB, API, 파일)가 바뀌면 캐시가 구버전 정보를 계속 줄 수 있다. 이때 무효화를 하지 않으면 **stale data(오래된 데이터)** 문제가 발생한다.

대표적인 방식은 다음과 같다.

- **TTL(Time To Live)**: 일정 시간이 지나면 자동 만료
- **Write-through / Write-back**: 쓰기 시점에 캐시와 원본 동기화
- **Cache Aside**: 애플리케이션이 캐시를 직접 읽고, 없으면 DB 조회 후 저장
- **Event-based invalidation**: 데이터 변경 이벤트가 발생하면 관련 캐시 삭제

핵심은 **“언제 틀렸다고 판단할 것인가”**이다. 너무 자주 지우면 캐시 히트율이 떨어지고, 너무 늦게 지우면 정확성이 깨진다. 그래서 실무에서는 데이터 변경 빈도, 일관성 요구 수준, 트래픽 패턴을 함께 고려한다.

## 짧은 예시 / 비유

예를 들어 상품 가격을 캐시에 10분 저장했다고 하자.  
관리자가 1분 뒤 가격을 수정했는데 캐시를 갱신하지 않으면, 사용자는 9분 동안 옛 가격을 보게 된다.

비유하면 **냉장고에 붙여둔 식단표**와 같다. 메뉴가 바뀌었는데 메모를 수정하지 않으면 계속 틀린 정보를 보고 장을 보게 된다.

## 면접 질문

**Q1. 캐시 무효화가 왜 어려운가요?**  
A. 캐시는 빠른 대신 원본과 분리되어 있어, 변경 시점을 정확히 맞추기 어렵기 때문이다.

**Q2. TTL만으로 충분한가요?**  
A. 아니다. 단순하지만 즉시 반영이 안 되므로, 정확성이 중요하면 이벤트 기반 삭제나 쓰기 시 동기화가 필요하다.

## 한 줄 요약

**캐시 무효화는 “빠름”과 “정확함” 사이의 균형을 맞추는 핵심 설계다.**
