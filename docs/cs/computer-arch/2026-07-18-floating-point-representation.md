# 부동소수점 표현과 실무에서의 오차 처리

- 실수는 대부분 이진수로 정확히 표현되지 않으므로 `0.1 + 0.2 != 0.3` 같은 오차가 발생한다.
- IEEE 754는 부호, 지수, 가수로 값을 저장하며, `float64`도 모든 실수를 무한 정밀도로 표현하지 못한다.
- 금액·수량·비교 로직에서는 부동소수점을 그대로 사용하지 말고 정수 단위, 고정소수점, `Decimal` 등을 선택해야 한다.

## 개념 설명

컴퓨터의 부동소수점은 일반적으로 IEEE 754 표준을 따른다. 값은 `부호 × 가수 × 2^지수` 형태로 저장된다. `float32`는 32비트, `float64`는 64비트를 사용하며, 비트 수가 늘어나면 표현 가능한 정밀도와 범위가 커진다. 하지만 유한한 이진 비트로 무한히 많은 실수를 표현할 수는 없다.

특히 십진수 `0.1`은 이진수로 무한히 반복되므로 저장 시 가장 가까운 값으로 반올림된다. 여러 연산이 누적되면 오차도 누적된다. 따라서 실수의 동등 비교는 `==`보다 허용 오차를 사용하는 방식이 안전하다. 단, 매우 큰 수와 작은 수를 함께 계산하면 작은 값이 사라지는 흡수 현상도 발생할 수 있다.

실무에서 센서 데이터, 그래픽 좌표, 물리 시뮬레이션, 머신러닝에서는 작은 오차를 허용하는 `float32` 또는 `float64`가 효율적이다. 반면 결제 금액, 세금, 포인트처럼 결과가 법적·재무적으로 정확해야 하는 영역에서는 금액을 원 단위 정수로 저장하거나 `Decimal`을 사용한다. 데이터베이스의 `FLOAT`와 `DECIMAL` 선택도 같은 기준으로 결정한다.

비교 시 절대 오차만 사용하면 값의 크기에 따라 부적절할 수 있다. 일반적으로 절대 오차와 상대 오차를 함께 고려한다. 또한 합산이 많은 경우 작은 값부터 더하거나, 오차 보정 알고리즘을 적용하면 정밀도를 개선할 수 있다. 직렬화 시에는 불필요하게 긴 소수 표현을 API에 노출하지 말고 명확한 반올림 정책을 정의해야 한다.

```python
from decimal import Decimal
from math import isclose

x = 0.1 + 0.2
print(x == 0.3)                    # False
print(isclose(x, 0.3, rel_tol=1e-12))

price = Decimal("19.99")
tax = Decimal("0.10")
total = price * (Decimal("1") + tax)
print(total.quantize(Decimal("0.01")))  # 21.99
```

## 면접 질문

### 1. `0.1 + 0.2`가 `0.3`과 같지 않을 수 있는 이유는?

`0.1`과 `0.2`가 이진수로 정확히 표현되지 않아 근사값으로 저장되기 때문이다. 연산 결과도 근사값이므로 동등 비교 대신 허용 오차 비교를 사용한다.

### 2. 금융 시스템에서 `float` 대신 무엇을 사용하는가?

가장 작은 화폐 단위의 정수로 저장하거나 `DECIMAL`·`BigDecimal`을 사용한다. 반올림 모드와 소수 자릿수를 명시해 계산 규칙을 일관되게 유지한다.

**한 줄 정리:** 부동소수점은 빠르고 넓은 범위를 제공하지만, 정확성이 필요한 값에는 표현 방식과 비교·반올림 정책을 별도로 설계해야 한다.
