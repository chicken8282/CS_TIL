# 캐시 계층(Cache Hierarchy)과 성능 최적화

- **핵심은 데이터 접근 시간 차이**다: CPU는 매우 빠르지만 메모리는 상대적으로 느려서, 가까운 저장소를 계층적으로 배치한다.
- **지역성(Locality)이 성능을 좌우**한다: 자주 쓰는 데이터와 가까운 코드/데이터를 유지하면 캐시 적중률이 올라간다.
- **실무 최적화 포인트가 명확하다**: 배열 순회, 데이터 배치, 반복문 구조, 객체 크기 관리가 캐시 효율에 큰 영향을 준다.

## Concept explanation

캐시 계층은 보통 **CPU 레지스터 → L1 캐시 → L2 캐시 → L3 캐시 → 메인 메모리(RAM) → 디스크** 순서로 구성된다. 위로 갈수록 빠르지만 작고 비싸며, 아래로 갈수록 느리지만 크고 저렴하다.  
성능 최적화 관점에서 중요한 것은 **CPU가 다음에 필요로 할 데이터를 미리 가까운 곳에 두는 것**이다. 캐시는 보통 **캐시 라인(cache line)** 단위로 데이터를 가져오므로, 한 번 읽을 때 연속된 데이터가 함께 들어오면 효율이 좋다.

최적화에 자주 등장하는 개념은 다음과 같다:
- **시간 지역성**: 최근 사용한 데이터는 곧 다시 사용될 가능성이 높다.
- **공간 지역성**: 인접한 데이터도 곧 사용할 가능성이 높다.
- **캐시 미스**: 원하는 데이터가 캐시에 없어 더 느린 상위 계층에서 가져오는 상황.
- **캐시 친화적 코드**: 연속 메모리 접근, 불필요한 분기 최소화, 구조체 정렬 등이 포함된다.

실무에서는 `O(n)`이라도 **메모리 접근 패턴**에 따라 체감 성능이 크게 달라진다. 예를 들어 2차원 배열을 행 기준으로 순회하면 캐시 적중률이 높지만, 열 기준 순회는 미스가 늘어 느려질 수 있다.

## Short example or analogy

도서관으로 비유하면, **책상 위 자료 = L1**, **내 서랍 = L2**, **사무실 책장 = L3**, **도서관 본관 = RAM**이다.  
자주 보는 문서를 책상 위에 두면 빨리 꺼낼 수 있지만, 안 쓰는 책을 전부 책상에 올려두면 오히려 정리가 안 된다.  
즉, **자주 쓰는 데이터만 가까이 두는 전략**이 캐시 최적화의 핵심이다.

예:
```text
좋음: for i in range(n): sum += arr[i]   # 연속 접근
나쁨: for i in range(n): sum += arr[idx[i]]  # 산발적 접근
```

## Interview questions

**Q1. 캐시 계층이 왜 필요한가요?**  
A1. CPU와 메모리 속도 차이를 줄여 평균 접근 시간을 낮추기 위해서입니다.

**Q2. 캐시 성능을 높이는 방법은?**  
A2. 지역성을 활용해 연속 메모리 접근을 늘리고, 불필요한 분기와 랜덤 접근을 줄이면 됩니다.

## Takeaway

**캐시 최적화는 “빠른 CPU를 더 빨리 일하게 만드는 데이터 배치 문제”다.**
