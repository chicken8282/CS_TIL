# CPU 파이프라인

## 핵심 요약

- CPU 명령어를 `Fetch → Decode → Execute → Memory → Write Back` 같은 단계로 나누어 여러 명령어를 동시에 처리한다.
- 파이프라인은 **한 명령어의 지연 시간(latency)**보다 전체 처리량(throughput)을 높이는 기법이다.
- 데이터 의존성, 분기, 메모리 지연으로 인해 `stall`, `flush`, `forwarding`이 발생할 수 있다.

## 개념 설명

파이프라인이 없는 CPU는 한 명령어를 끝낸 뒤 다음 명령어를 시작한다. 반면 파이프라인 CPU는 명령어마다 작업을 여러 단계로 나누고, 각 단계가 서로 다른 명령어를 동시에 처리한다. 5단계 파이프라인에서는 첫 명령어가 완료되기까지 5사이클이 필요하지만, 파이프라인이 채워진 뒤에는 이상적으로 매 사이클 하나의 명령어가 완료된다.

일반적인 단계는 다음과 같다.

1. **IF(Fetch)**: 프로그램 카운터가 가리키는 명령어를 가져온다.
2. **ID(Decode)**: 명령어를 해석하고 레지스터를 읽는다.
3. **EX(Execute)**: ALU 연산이나 주소 계산을 수행한다.
4. **MEM**: 데이터 메모리에 접근한다.
5. **WB(Write Back)**: 결과를 레지스터에 기록한다.

파이프라인의 성능은 단계 수만으로 결정되지 않는다. 가장 느린 단계의 시간이 전체 사이클 시간을 결정하며, 레지스터 저장과 제어 로직의 오버헤드도 추가된다. 따라서 단계가 많으면 클록을 높일 수 있지만, 분기 실패 시 버려야 할 명령어가 늘어날 수 있다.

대표적인 위험은 세 가지다. **구조적 위험**은 여러 명령어가 같은 하드웨어를 요구할 때 발생한다. **데이터 위험**은 앞 명령어의 결과를 뒤 명령어가 즉시 사용할 때 발생한다. **제어 위험**은 조건 분기 결과를 알기 전에 다음 명령어를 예측해야 할 때 발생한다.

데이터 위험은 결과를 레지스터에 기록할 때까지 기다리는 대신, 실행 결과를 다음 단계로 직접 전달하는 **forwarding**으로 줄일 수 있다. 해결할 수 없는 경우에는 파이프라인을 멈추는 **stall**이 필요하다. 분기 예측이 틀리면 이미 가져온 명령어를 제거하는 **flush**가 발생한다.

## 코드 예제

```c
int dependent(int x) {
    int a = x + 1;
    int b = a * 2;      // a 결과에 의존: stall 또는 forwarding
    int c = b + 3;      // b 결과에 의존
    return c;
}

int independent(int x, int y) {
    int a = x + 1;
    int b = y * 2;      // a와 독립적: 병렬 실행 가능성
    return a + b;
}
```

`dependent`는 앞 연산 결과가 다음 연산의 입력이므로 데이터 위험이 연속해서 발생한다. `independent`는 두 계산이 서로 독립적이어서 슈퍼스칼라 CPU가 여러 실행 유닛에 나누어 처리할 가능성이 높다. 단, 실제 결과는 컴파일러 최적화, CPU의 실행 폭, 레지스터 할당에 따라 달라진다.

## 면접 질문

### 1. 파이프라인에서 forwarding과 stall의 차이는?

**답변:** Forwarding은 아직 레지스터에 기록되지 않은 결과를 다음 명령어에 직접 전달해 대기를 줄인다. Forwarding으로도 해결할 수 없는 의존성이면 파이프라인을 멈추는 stall을 사용한다.

### 2. 분기 예측 실패 시 CPU는 무엇을 하는가?

**답변:** 잘못 예측해 파이프라인에 들어온 명령어를 flush하고, 올바른 분기 주소에서 명령어를 다시 가져온다. 파이프라인이 깊을수록 손실 사이클이 커질 수 있다.

## 한 줄 정리

CPU 파이프라인은 명령어를 단계적으로 겹쳐 처리해 처리량을 높이지만, 의존성과 분기 때문에 forwarding·stall·flush가 필요하다.
