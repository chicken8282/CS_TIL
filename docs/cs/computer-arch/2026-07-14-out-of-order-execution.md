# Out-of-Order Execution

- CPU가 **명령어를 프로그램 순서와 다르게 실행**해도, 최종적으로는 **순서대로 실행한 것처럼 보이게** 만드는 최적화 기법이다.
- 핵심 목적은 **파이프라인 정체를 줄여 ILP(Instruction-Level Parallelism)** 를 극대화하는 것이다.
- 실무/면접에서는 **“성능 최적화”와 “메모리 순서 보장”의 긴장 관계**를 함께 이해해야 한다.

## Concept explanation

Out-of-order execution(OoO)는 앞선 명령이 메모리 대기, 캐시 미스, 분기 지연 등으로 멈춰 있을 때, 뒤에 있는 독립적인 명령을 먼저 실행하는 방식이다.  
예를 들어 `A -> B -> C` 순서로 코드가 있어도, `A`가 메모리에서 값을 기다리는 동안 `C`가 `A/B`와 독립적이면 먼저 실행할 수 있다. 실행은 뒤섞여도 **커밋(commit)** 은 원래 순서를 유지해 프로그램 의미를 보장한다.

자주 하는 실수:
- **“CPU가 순서를 바꾸면 프로그램 결과도 바뀐다”**고 단정하기  
  → 일반적인 단일 스레드에서는 커밋 단계가 순서를 보장한다.
- **성능 문제를 무조건 OoO 탓으로 돌리기**  
  → 실제 병목은 캐시 미스, 분기 예측 실패, false sharing, 락 경합일 수 있다.
- **동기화 없이 멀티스레드 메모리 가시성을 기대하기**  
  → OoO와 컴파일러 재배치 때문에 한 스레드의 쓰기가 다른 스레드에 즉시/순서대로 보인다고 가정하면 안 된다.
- **volatile을 만능으로 오해하기**  
  → 일부 재배치 억제는 가능하지만, 원자성/복합 연산/전체 동기화는 보장하지 않는다.

면접 포인트: OoO는 **실행 순서**를 바꾸지만, **아키텍처 상태**는 리오더 버퍼(ROB) 등으로 정리해 순차적 의미를 유지한다.

## Code example

```c
// 멀티스레드에서 "쓴 순서대로 보인다"를 가정하면 위험
int data = 0;
int ready = 0;

// Thread 1
data = 42;
ready = 1;

// Thread 2
if (ready == 1) {
    printf("%d\n", data); // 42가 아닐 수도 있음(가시성/재배치 문제)
}
```

## Interview questions

1. **Q. OoO execution이 왜 성능을 높이나요?**  
   A. 앞선 명령의 대기를 숨기고 독립 명령을 먼저 처리해 CPU 실행 유휴 시간을 줄이기 때문입니다.

2. **Q. OoO 때문에 멀티스레드 버그가 생기는 이유는?**  
   A. 실행/메모리 순서가 재배치될 수 있어, 동기화 없이 다른 스레드가 중간 상태를 관찰할 수 있기 때문입니다.

**Takeaway:** OoO는 CPU의 성능 최적화지만, 개발자는 “순서대로 보인다”를 믿기 전에 반드시 메모리 모델과 동기화를 점검해야 한다.
