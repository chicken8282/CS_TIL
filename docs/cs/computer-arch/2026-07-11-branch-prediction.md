# Branch Prediction

- 분기 예측은 CPU가 **조건문 결과를 미리 추측**해 파이프라인을 계속 채우는 최적화다.
- 내부적으로는 **BTB(Branch Target Buffer), BHT/예측 비트, 스택(RAS)** 같은 구조로 “다음에 어디로 갈지”를 판단한다.
- 예측이 틀리면 **파이프라인 flush + 재실행** 비용이 커서, 조건문 패턴이 성능에 큰 영향을 준다.

## Concept explanation

현대 CPU는 명령을 한 번에 하나씩 실행하지 않고, 여러 단계를 겹쳐 처리하는 **파이프라인**을 사용한다. 그런데 `if`, `for`, `while` 같은 분기에서 다음 주소를 모르면 미리 가져온 명령이 헛될 수 있다. 그래서 CPU는 분기가 **taken(점프)** 인지 **not taken(순차 진행)** 인지 추측한다.

내부 구현은 보통 다음 요소로 구성된다.

- **BTB**: 과거에 분기했던 주소와 목적지 주소를 저장한다. “이 분기면 어디로 점프했는가”를 빠르게 찾는다.
- **BHT / 예측 테이블**: 분기 주소의 일부 비트를 인덱스로 삼아, 1-bit 또는 2-bit saturating counter로 taken 여부를 예측한다. 2-bit 방식은 한 번의 이상치에 덜 흔들린다.
- **RAS(Return Address Stack)**: `call/ret`에 특화된 스택으로, 함수 복귀 주소를 예측한다.

예측이 맞으면 파이프라인이 계속 진행되어 성능이 좋아진다. 반대로 틀리면 이미 가져온 명령을 버리고(correct path로) 다시 시작해야 하므로, 분기 미스패널티가 발생한다. 따라서 **예측 가능한 분기 패턴**이 중요하다. 예를 들어, 데이터 의존적 `if`가 불규칙하면 미스가 늘고, 반복문처럼 패턴이 일정하면 예측률이 높다.

## Short example or analogy

자판기에서 매번 버튼을 누르기 전에 “다음에 음료가 나올 것”이라고 미리 트레이를 준비하는 것과 비슷하다. 맞으면 빠르지만, 틀리면 준비한 트레이를 치우고 다시 준비해야 한다. CPU도 같은 방식으로 “다음 명령”을 미리 준비한다.

```c
if (x > 0) sum += x;
else sum -= x;
```

`x > 0`의 분포가 한쪽으로 치우치면 예측이 잘 맞는다. 하지만 랜덤하면 매번 추측이 흔들려 성능이 떨어진다.

## Interview questions

**Q1. 왜 분기 예측이 성능에 중요한가?**  
A1. 분기 결과를 기다리면 파이프라인이 멈추기 때문이다. 예측이 맞으면 실행을 끊지 않고 계속 처리할 수 있다.

**Q2. 2-bit 예측기가 1-bit보다 유리한 이유는?**  
A2. 한 번의 예외 상황에 바로 상태가 뒤집히지 않아, 반복 패턴에 더 안정적으로 대응하기 때문이다.

## Takeaway

분기 예측은 “조건문의 미래를 맞히는 하드웨어 캐시”이며, 예측 정확도가 곧 CPU 성능이다.
