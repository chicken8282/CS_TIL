# CPU Pipeline

- CPU 파이프라인은 명령어를 **여러 단계로 쪼개 겹쳐 처리**해 처리량을 높이는 구조다.
- 실무에서는 **분기 예측 실패, 캐시 미스, 의존성(stall)** 때문에 이론 성능보다 훨씬 느려질 수 있다.
- 튜닝 포인트는 **분기 감소, 메모리 접근 패턴 개선, 루프/배치 최적화**다.

## Concept explanation

CPU 파이프라인은 보통 `Fetch → Decode → Execute → Memory → Writeback` 단계로 나뉜다.  
핵심은 한 명령어가 실행되는 동안 다음 명령어를 미리 가져와서, 여러 명령어를 동시에 각 단계에 배치하는 것이다. 이렇게 하면 **처리량(throughput)** 은 증가하지만, **한 개 명령어의 지연시간(latency)** 이 항상 줄어드는 것은 아니다.

실무 트러블슈팅에서 중요한 건 “CPU 사용률이 높은데도 속도가 안 나는 이유”를 파이프라인 관점으로 보는 것이다. 대표적으로:
- **분기 예측 실패**: `if`가 자주 바뀌는 코드, 예측 어려운 패턴은 pipeline flush를 유발
- **데이터 의존성**: 이전 계산 결과가 나와야 다음 계산이 가능하면 stall 발생
- **메모리 병목**: RAM 접근이 느리면 실행 유닛이 놀게 됨
- **명령어 캐시/데이터 캐시 미스**: 연속성이 낮은 접근은 파이프라인 효율 저하

즉, “연산 자체는 가벼운데 느리다”면 알고리즘뿐 아니라 **CPU가 기다리는 이유**를 봐야 한다.

## Short example or analogy

비유하면 라면 공장이다.  
한 사람이 한 그릇을 처음부터 끝까지 만들면 느리다.  
면 삶기, 스프 넣기, 토핑 올리기를 각 담당자가 나눠서 동시에 처리하면 **완성 속도**가 빨라진다.  
하지만 한 단계에서 재료가 늦게 오면 뒤 공정이 모두 멈춘다. CPU도 마찬가지로, 한 명령어의 지연이나 분기 실패가 전체 흐름을 막는다.

## Interview questions

**Q1. CPU pipeline의 장점은 무엇인가요?**  
A. 여러 명령어를 겹쳐 처리해 **처리량을 높인다**. 같은 시간에 더 많은 일을 처리할 수 있다.

**Q2. 실무에서 pipeline 성능이 떨어지는 대표 원인은?**  
A. **분기 예측 실패, 캐시 미스, 데이터 의존성**이다. 특히 조건문이 많거나 메모리 접근이 불규칙한 코드에서 자주 발생한다.

**Takeaway:** 파이프라인은 “빠른 CPU”의 핵심이지만, 실무 성능은 결국 **stalls를 얼마나 줄이느냐**에 달려 있다.
