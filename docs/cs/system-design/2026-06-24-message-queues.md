# 메시지 큐(Message Queue)

- 비동기 처리를 위해 생산자와 소비자를 분리한다.
- 트래픽 폭주를 흡수해 시스템 안정성을 높인다.
- 재시도, 지연 처리, 작업 분산에 자주 쓰인다.

## 개념 설명

메시지 큐는 **한 시스템이 메시지를 큐에 넣고(produce), 다른 시스템이 나중에 꺼내 처리(consume)** 하는 통신 방식이다.  
요청을 즉시 처리하지 않고 중간 저장소에 쌓아두기 때문에, 서비스 간 결합도를 낮추고 장애 전파를 줄일 수 있다.

대표적인 흐름은 다음과 같다.

1. 주문 서비스가 “주문 생성” 메시지를 큐에 넣음  
2. 알림 서비스, 결제 서비스, 재고 서비스가 각자 큐 또는 토픽에서 메시지를 읽음  
3. 각 소비자는 독립적으로 처리 속도를 조절함

장점:
- **비동기 처리**: API 응답을 빠르게 끝내고 뒤에서 작업 수행
- **확장성**: 소비자 수를 늘려 처리량 증가
- **내결함성**: 한 소비자가 죽어도 메시지는 남아 재처리 가능

주의점:
- **중복 처리** 가능성: at-least-once 전달에서는 같은 메시지가 여러 번 올 수 있음
- **순서 보장**: 파티션/키 설계를 잘해야 함
- **지연 시간**: 즉시성보다 처리량에 유리

실무에서는 RabbitMQ, Kafka, SQS 등을 많이 사용한다.  
RabbitMQ는 라우팅과 작업 큐에 강하고, Kafka는 대용량 이벤트 스트리밍과 로그 적재에 강하다.

## 짧은 예시 / 비유

비유하자면, 메시지 큐는 **식당의 주문 대기표**와 같다.  
손님(생산자)은 주문만 접수하면 되고, 주방(소비자)은 자기 속도대로 주문을 처리한다. 손님이 몰려도 대기표가 있으니 주문이 사라지지 않는다.

예시:
- 회원가입 직후 이메일 발송
- 이미지 썸네일 생성
- 결제 완료 후 포인트 적립

## 면접 질문

**Q1. 메시지 큐를 왜 쓰나요?**  
A. 서비스 간 직접 호출을 줄여 결합도를 낮추고, 비동기 처리로 응답 속도와 확장성을 높이기 위해서입니다.

**Q2. 중복 메시지는 어떻게 처리하나요?**  
A. 소비자에서 멱등성을 보장하거나, 메시지 ID를 저장해 이미 처리한 요청은 무시합니다.

## 한 줄 정리

메시지 큐는 **시스템을 느슨하게 연결하고, 트래픽과 장애를 흡수하는 비동기 통신 도구**다.
